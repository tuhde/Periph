# Chip Spec: PCF8575 (IO Expander)

**Manufacturer:** NXP (formerly Philips Semiconductors)  
**Datasheet:** `datasheets/io_expander/pcf8575.pdf`  
**Category:** `io_expander`  
**Transports:** I²C

## Overview

The PCF8575 is a 16-bit quasi-bidirectional I/O port expander that connects to a host via I²C (Fast mode, 400 kHz). Its 16 pins are organised as two independent 8-bit ports: Port 0 (P00–P07) and Port 1 (P10–P17). Like the PCF8574, each pin is quasi-bidirectional with no separate direction register — writing a 1 enables the internal 100 µA current source (input/high-impedance mode) while writing a 0 drives the pin strongly low (open-drain output). Reading always returns the actual logic level. An open-drain active-low INT output is asserted whenever any input pin changes state. Every I²C transaction transfers exactly two bytes — Port 0 first, Port 1 second — making the protocol slightly different from the 8-bit PCF8574.

Up to 8 devices can be connected on the same bus using the three hardware address pins A0, A1, A2 (all giving `0x20`–`0x27`; there is no PCF8575A variant).

## Transport Configuration

### I²C
- **Address:** `0x20` (default, A2=A1=A0=0) — `0x21`–`0x27` (A2, A1, A0 combinations)
- **Max clock:** 400 kHz (Fast mode)

## Register Map

The PCF8575 has no sub-registers. Every transaction is a 2-byte sequence:

| Transaction | Bytes | Description |
|-------------|-------|-------------|
| Read  | 2 bytes received | Byte 1 = actual logic levels P07–P00; Byte 2 = P17–P10 |
| Write | 2 bytes sent     | Byte 1 = output latch P07–P00; Byte 2 = output latch P17–P10 |

|------|------|-------------|

| 3 | P03 | Pin 3 of Port 0 |

## Pin Capabilities

| Capability | Support |
|-----------|---------|
| Pin count | 16 (P00–P07, P10–P17) |
| Ports | 2 × 8-bit |
| Direction | quasi-bidirectional per-pin — implicit: write 1 = input, write 0 = output low |
| Pull-up | yes — internal ~100 µA current source when pin written 1 |
| Pull-down | no |
| Open-drain output | yes — pin written 0 sinks up to 25 mA; written 1 is high-impedance |
| Push-pull output | no — cannot actively drive high |
| Interrupt output | yes — active-low open-drain INT, asserted on any input change (rising or falling) |
| Interrupt modes | any-change on input pins only; cleared by any read or write to the device |
| Drive strength | no |

## Initialization Sequence

1. Write `[0xFF, 0xFF]` to the device — sets all 16 pins to quasi-bidirectional input mode with internal pull-ups.
2. Store `[0xFF, 0xFF]` in the driver's shadow registers (one byte per port).

No delay is required between power-on and the first transaction.

## Implementation Stages

Each chip is implemented in two stages. The Full class extends Minimal.

The defining characteristic of an IO expander driver is that it exposes individual **Pin objects** that implement each platform's native GPIO interface — see the GPIO Interface section below. Users obtain a pin from the driver and use it exactly like a hardware GPIO pin.

### Minimal

Goal: expose all 16 pins as GPIO objects with direction control and value read/write. No interrupt support required.

**Driver API**

| Operation | Parameters | Returns | Notes |
|-----------|------------|---------|-------|
| `__init__` | `transport`, `addr=0x20` | — | Write `[0xFF, 0xFF]`; initialise shadow registers to `[0xFF, 0xFF]` |
| `pin` | `n: int` | Pin | Return a Pin proxy for pin number `n` (0–15); port = n // 8, bit = n % 8 |
| `read_port` | `port: int` | int | Read the 8 pins of the given port (0 or 1) as a bitmask; always performs a full 2-byte I²C read, returns the requested byte |
| `write_port` | `port: int`, `mask: int` | — | Write all 8 pins of the given port at once; preserves the other port's shadow value; updates shadow register |

**Sensible defaults:** all 16 pins as quasi-bidirectional inputs; shadow registers = `[0xFF, 0xFF]`; interrupts disabled.

**I²C wire protocol note:** The device always sends/receives exactly 2 bytes. `read_port` always performs a 2-byte read and returns the byte for the requested port. `write_port` always performs a 2-byte write using the shadow value for the untouched port.

**Pin API — Minimal**

| Operation | MicroPython | CircuitPython | C++ | Node.js | Rust |
|-----------|-------------|---------------|-----|---------|------|
| Get pin | `chip.pin(n)` | `chip.pin(n)` | `chip.pin(n)` | `chip.pin(n, 'in')` | `chip.pin(n)` |
| Set direction | `pin.init(Pin.IN)` / `pin.init(Pin.OUT)` | `pin.direction = Direction.INPUT` / `Direction.OUTPUT` | `pin.mode(INPUT)` / `pin.mode(OUTPUT)` | `pin.setDirection('in', cb)` / `'out'` | `chip.input_pin(n)` / `chip.output_pin(n)` |
| Set high (→ input mode) | `pin.on()` | `pin.value = True` | `pin.high()` | `pin.writeSync(1)` | `pin.set_high()?` |
| Set low | `pin.off()` | `pin.value = False` | `pin.low()` | `pin.writeSync(0)` | `pin.set_low()?` |
| Read | `pin.value()` | `pin.value` | `pin.read()` | `pin.readSync()` | `pin.is_high()?` |
| Toggle | `pin.toggle()` | *(manual)* | `pin.toggle()` | *(manual)* | *(manual)* |
| Release | — | `pin.deinit()` | — | `pin.unexport()` | *(drop)* |

### Full

Goal: expose interrupt-on-change functionality. Extends Minimal.

**Driver API additions**

| Operation | Parameters | Returns | Notes |
|-----------|------------|---------|-------|
| *(inherits Minimal)* | | | |
| `configure_interrupt` | `int_pin`, `callback` | — | Attach `callback(changed_mask: int)` to the chip's INT line; `changed_mask` is a 16-bit value (bits 0–7 = Port 0, bits 8–15 = Port 1). `int_pin` is a hardware GPIO pin pre-configured as input (MicroPython `machine.Pin`, Linux `gpiod` line, or equivalent). On Linux, delivers interrupts via a polling thread when no hardware IRQ is available (pass `int_pin=None`). |
| `clear_interrupt` | — | int | Read all 16 pins (2-byte I²C read), compare to previous read, return 16-bit bitmask of pins that changed; clears the INT line. |

**Pin API additions — Full**

| Operation | MicroPython | CircuitPython | C++ | Node.js | Rust |
|-----------|-------------|---------------|-----|---------|------|
| Attach interrupt | `pin.irq(handler, Pin.IRQ_FALLING)` | *(via driver)* | `pin.attachInterrupt(handler, FALLING)` | `pin.watch(handler)` | *(via driver callback)* |
| Detach interrupt | `pin.irq(None)` | *(via driver)* | `pin.detachInterrupt()` | `pin.unwatch(handler)` | *(via driver)* |

**Additional configuration options:** none beyond what Minimal provides — the PCF8575 has no interrupt mask, polarity, or trigger-mode registers.

## GPIO Interface

This section defines the per-platform contracts for Pin objects. See `AGENTS.md` for implementation patterns.

### Python — MicroPython

Pin objects implement a compatible subset of `machine.Pin`. Constants (`IN`, `OUT`, `IRQ_FALLING`, `IRQ_RISING`) are defined as class attributes on `Pcf8575Minimal`.

Minimal interface: `init(mode)`, `value([x])`, `on()`, `off()`, `toggle()`  
Full adds: `irq(handler, trigger)` — trigger is `IRQ_FALLING` or `IRQ_RISING` (the PCF8575 INT is any-edge; `IRQ_RISING` re-fires on the next read that shows the pin returned high).

### Python — CircuitPython

Pin objects implement `digitalio.DigitalInOut`.

Minimal interface: `direction` (read/write), `value` (read/write), `switch_to_input()`, `switch_to_output(value=False)`  
Full adds: no additional properties (`pull` and `drive_mode` are not applicable — the PCF8575 has a fixed internal pull-up and open-drain output). Both setters should raise `AttributeError` with a descriptive message.

### Python — Linux

Same interface as MicroPython (`init`, `value`, `on`, `off`, `toggle`). Interrupt delivery in Full uses a background polling thread that calls `clear_interrupt()` every 5 ms; when a change is detected the callback is invoked with the 16-bit changed mask. If a `gpiod` line for the INT pin is passed to `configure_interrupt`, use edge detection instead of polling.

### C++

`IOExpanderPin` is a nested class in `Pcf8575Minimal`. Arduino GPIO constants are reused (`INPUT`, `OUTPUT`, `HIGH`, `LOW`).

Minimal interface:
```cpp
class IOExpanderPin {
public:
    void mode(uint8_t m);     // INPUT, OUTPUT
    void write(uint8_t v);    // HIGH (→ input mode), LOW (→ drive low)
    uint8_t read();
    void high()   { write(HIGH); }
    void low()    { write(LOW); }
    void toggle() { write(read() ^ 1); }
};
```

Full adds: `attachInterrupt(void (*handler)(uint16_t changed_mask), uint8_t mode)` and `detachInterrupt()` to `Pcf8575Full`. `mode` accepts `FALLING` only (the INT pin is active-low; any-change is resolved in software by comparing reads). On Linux, deliver via a `poll()` thread on the INT GPIO sysfs fd. On Zephyr, deliver via `gpio_add_callback()`. The `changed_mask` is a 16-bit value: bits 0–7 = Port 0, bits 8–15 = Port 1.

### Node.js

Pin objects implement the [`onoff`](https://www.npmjs.com/package/onoff) `Gpio` interface subset.

Minimal interface: `readSync()`, `writeSync(value)`, `read(callback)`, `write(value, callback)`, `direction` property, `unexport()`  
Full adds: `watch(callback)`, `unwatch(callback)` — interrupt delivery via polling the INT GPIO using `epoll` (Linux sysfs) or via a 5 ms poll interval when no INT line is available.

### Rust

Output pins implement `embedded_hal::digital::OutputPin`; input pins implement `embedded_hal::digital::InputPin`. The driver wraps its I2C bus in `core::cell::RefCell<I2C>`.

The `pin()` factory returns an `ExPin<'_, I2C>` that implements both traits, since the PCF8575 does not distinguish direction at the register level. Callers that need a typed pin use `input_pin(n)` → `InputExPin` or `output_pin(n)` → `OutputExPin` — thin wrappers that only expose the relevant trait.

Full adds `embedded_hal::digital::StatefulOutputPin`: reading back the shadow register (not the bus) reports what was last written to the output latch.

Re-export from category `mod.rs` (note: `ExPin` must be disambiguated from the PCF8574 re-export):
```rust
pub use pcf8575::{Pcf8575Minimal, Pcf8575Full, ExPin as Pcf8575ExPin};
```
The existing PCF8574 re-export should be updated to:
```rust
pub use pcf8574::{Pcf8574Minimal, Pcf8574Full, ExPin as Pcf8574ExPin};
```

### JVM

The `Pin` class is a `public static class` nested inside `Pcf8575Minimal`. Interrupt delivery is always via a background polling thread; no hardware GPIO edge detection is used.

Minimal interface:
```java
chip.pin(n)          // → Pcf8575Minimal.Pin  (n = 0–15)
pin.setInput()       // write 1 (release to quasi-input)
pin.setOutput()      // write 0 (drive low)
pin.setHigh()        // write 1 — alias for setInput
pin.setLow()         // write 0 — alias for setOutput
pin.read()           // → boolean (bus read, not shadow)
pin.toggle()         // invert shadow bit
```

Full adds interrupt at the driver level:
```java
chip.configureInterrupt(IntConsumer callback)  // start 5 ms polling thread; callback receives 16-bit changed mask
chip.clearInterrupt()                          // → int 16-bit changed bitmask; clears INT
chip.stopInterrupt()                           // stop polling thread
```

In Kotlin, `configureInterrupt` accepts a `(Int) -> Unit` lambda. In Groovy, it accepts a `Closure`.  
No per-pin interrupt methods exist in the JVM interface; per-pin callbacks should be dispatched inside the driver-level callback using the changed bitmask.

## Data Conversion

```
pin n  →  port = n // 8,  bit = n % 8

Read pin n:
    [b0, b1] = read 2 bytes
    byte = b0 if port == 0 else b1
    value = (byte >> bit) & 1

Write pin n:
    port_idx = n // 8
    if value == 1:  shadow[port_idx] |= (1 << bit)
    else:           shadow[port_idx] &= ~(1 << bit)
    write 2 bytes: [shadow[0], shadow[1]]
```

Port 0 shadow register covers pins 0–7; Port 1 shadow register covers pins 8–15. The shadow registers are the authoritative record of what was last written. Both must be initialised to `0xFF` at construction and updated on every write operation.

Pin-to-port-byte mapping:
```
Pins  0– 7  →  Port 0 byte (P07=bit7 … P00=bit0)
Pins  8–15  →  Port 1 byte (P17=bit7 … P10=bit0)
```

### Demo flow

Eight inject nodes periodically write a rotating pattern to P00–P07 (LEDs on Port 0). Eight more inject nodes poll P10–P17 (buttons on Port 1) and send results to debug nodes. A final inject reads the full 16-bit value and displays both port bytes as hex strings. Shows simultaneous use of Port 0 write (LED control) and Port 1 read (button sense) with the 2-byte I²C protocol.

## Examples

### Demo

Wire eight LEDs (active-low, cathode to P00–P07, Port 0) and eight push buttons (P10–P17 to GND when pressed, floating high otherwise via internal pull-up). The demo mirrors button inputs to LEDs: P10→P00, P11→P01, …, P17→P07. It reads the full 16-bit value every 200 ms, extracts the Port 1 byte (bits 8–15), inverts it (button pressed = 0 → LED on = 0), and writes the result to Port 0 (bits 0–7). The `print` shows both the raw port bytes in hex and the decoded button/LED states so the quasi-bidirectional read-back behavior and 2-byte protocol are visible.

## Timing Constraints

- No startup delay required after power-on (power-on reset sets all I/Os high automatically).
- INT asserts within 4 µs after an input changes (t_iv ≤ 4 µs).
- INT resets within 4 µs after a read transaction (t_ir ≤ 4 µs).
- At 400 kHz, a full 2-byte read takes ~45 µs; a full 2-byte write takes ~45 µs.
- At 400 kHz, a read-modify-write cycle (read 2 bytes + write 2 bytes) takes ~90 µs.

## Implementation Notes

- **Two-byte protocol is mandatory.** Unlike the PCF8574 (1 byte), the PCF8575 always transfers 2 bytes. Sending or reading a single byte is protocol-incorrect; many I²C master implementations will NACK or return garbage. Always send/receive exactly 2 bytes.
- **No push-pull high drive.** Writing 1 enables the internal current source (~100 µA). This is sufficient for high-impedance inputs but cannot drive a low-impedance load high. LEDs must be wired active-low (cathode to pin, anode to VCC).
- **Shadow registers are mandatory.** The device has no read-back of the output latch. To set or clear individual bits, the driver must read the shadow (not the bus), modify it, and write both bytes.
- **INT is any-change on inputs only.** Pins with shadow bit = 0 (output mode, driven low) do not generate INT. Only pins with shadow bit = 1 (input mode) trigger INT when their state changes.
- **INT cleared by any access.** Any I²C read or write to the device clears the INT line. The driver must save the last-read value to detect which pins changed.
- **No PCF8575A variant.** Unlike the PCF8574 family, there is only one address range: `0x20`–`0x27`. No address overlap with common OLED/display addresses (`0x3C`, `0x3D`, etc.).
- **No reset pin.** The device resets only on power cycle; the internal power-on reset sets all I/Os to logic 1.
- **Fast mode (400 kHz).** The PCF8575 supports Fast mode I²C, unlike the PCF8574 which is Standard mode only (100 kHz). Implementations should default to 400 kHz.

## Sigrok Decoder

Decoder id `pcf8575`, input `['i2c']`. Matches addresses `0x20`–`0x27`. Annotates every transaction with both port bytes and the individual pin states P17–P10 (Port 1) and P07–P00 (Port 0). Distinguishes read (pin levels from bus) from write (output latch values). Each annotation shows the 16-bit hex value and the per-pin state as a two-group binary string (e.g. `P1=10110011 P0=11001100`). Emits a warning annotation for transactions that do not consist of exactly 2 data bytes.
