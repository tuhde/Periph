# Chip Spec: PCF8574 (IO Expander)

**Manufacturer:** Texas Instruments  
**Datasheet:** `datasheets/io_expander/pcf8574.pdf`  
**Category:** `io_expander`  
**Transports:** I²C

## Overview

The PCF8574 is an 8-bit quasi-bidirectional I/O port expander that connects to a host via I²C. Each of its eight pins (P0–P7) can independently source or sink current with no separate direction register: writing a 1 sets the pin to high-impedance input mode (a weak ~100 µA current source holds it high), while writing a 0 drives the pin strongly low. Reading always returns the actual logic level at the pin, regardless of what was written. An open-drain active-low INT output is asserted whenever any input pin changes state, making the PCF8574 well suited for button matrices, LED arrays, and general purpose GPIO expansion where push-pull high drive is not required.

Two address-range variants share identical register behavior:
- **PCF8574** — 7-bit address `0x20`–`0x27` (A2, A1, A0 strapped low by default → `0x20`)
- **PCF8574A** — 7-bit address `0x38`–`0x3F` (A2, A1, A0 strapped low by default → `0x38`)

## Transport Configuration

### I²C
- **Address:** `0x20` (default, A2=A1=A0=0) — `0x21`–`0x27` (PCF8574); `0x38`–`0x3F` (PCF8574A)
- **Max clock:** 100 kHz (standard mode only)

## Register Map

The PCF8574 exposes a single data byte. There are no sub-registers; every transaction to the device address reads or writes that byte.

| Access | Name | Reset | Description |
|--------|------|-------|-------------|
| R      | PORT_IN  | `0xFF` | Read actual logic levels of all 8 pins |
| W      | PORT_OUT | `0xFF` | Write output / direction latch for all 8 pins |

### Bit Fields

#### PORT byte

| Bits | Name | Description |
|------|------|-------------|
| 7    | P7   | Pin 7 — write 1: input (weak pull-up); write 0: drive low |
| 6    | P6   | Pin 6 |
| 5    | P5   | Pin 5 |
| 4    | P4   | Pin 4 |
| 3    | P3   | Pin 3 |
| 2    | P2   | Pin 2 |
| 1    | P1   | Pin 1 |
| 0    | P0   | Pin 0 |

## Pin Capabilities

| Capability | Support |
|-----------|---------|
| Pin count | 8 (P0–P7) |
| Ports | 1 × 8-bit |
| Direction | quasi-bidirectional per-pin — implicit: write 1 = input, write 0 = output low |
| Pull-up | yes — internal ~100 µA current source when pin written 1 |
| Pull-down | no |
| Open-drain output | yes — pin written 0 sinks up to 25 mA; written 1 is high-impedance |
| Push-pull output | no — cannot actively drive high |
| Interrupt output | yes — active-low open-drain INT, asserted on any input change |
| Interrupt modes | any-change on input pins only; cleared by reading PORT_IN |
| Drive strength | no |

## Initialization Sequence

1. Write `0xFF` to the device — sets all eight pins to quasi-bidirectional input mode with internal pull-ups.
2. Store `0xFF` in the driver's shadow register.

No delay is required between power-on and the first transaction.

## Implementation Stages

Each chip is implemented in two stages. The Full class extends Minimal.

The defining characteristic of an IO expander driver is that it exposes individual **Pin objects** that implement each platform's native GPIO interface — see the GPIO Interface section below. Users obtain a pin from the driver and use it exactly like a hardware GPIO pin.

### Minimal

Goal: expose all 8 pins as GPIO objects with direction control and value read/write. No interrupt support required.

**Driver API**

| Operation | Parameters | Returns | Notes |
|-----------|------------|---------|-------|
| `__init__` | `transport`, `addr=0x20` | — | Write 0xFF; initialise shadow register to 0xFF |
| `pin` | `n: int` | Pin | Return a Pin proxy for pin number `n` (0–7) |
| `read_port` | `port: int = 0` | int | Read all 8 pins as a bitmask; ignores `port` (only one port) |
| `write_port` | `port: int = 0`, `mask: int` | — | Write all 8 pins at once; updates shadow register |

**Sensible defaults:** all 8 pins as quasi-bidirectional inputs; shadow register = `0xFF`; interrupts disabled.

**Pin API — Minimal**

| Operation | MicroPython | CircuitPython | C++ | Node.js | Rust |
|-----------|-------------|---------------|-----|---------|------|
| Get pin | `chip.pin(n)` | `chip.pin(n)` | `chip.pin(n)` | `chip.pin(n, 'in')` | `chip.pin(n)` |
| Set direction | `pin.init(Pin.IN)` / `pin.init(Pin.OUT)` | `pin.direction = Direction.INPUT` / `Direction.OUTPUT` | `pin.mode(INPUT)` / `pin.mode(OUTPUT)` | `pin.setDirection('in', cb)` / `'out'` | `chip.input_pin(n)` / `chip.output_pin(n)` |
| Set high (→ input mode) | `pin.on()` | `pin.value = True` | `pin.high()` | `pin.writeSync(1)` | `pin.set_high()?` |
| Set low | `pin.off()` | `pin.value = False` | `pin.low()` | `pin.writeSync(0)` | `pin.set_low()?` |
| Read | `pin.value()` | `pin.value` | `pin.read()` | `pin.readSync()` | `pin.is_high()?` |
| Toggle | `pin.toggle()` | *(manual)* | `pin.toggle()` | *(manual)* | *(manual)* |
| Release | — | `pin.deinit()` | — | `pin.unexport()` | *(drop)* |

### Full

Goal: expose interrupt-on-change functionality. Extends Minimal.

**Driver API additions**

| Operation | Parameters | Returns | Notes |
|-----------|------------|---------|-------|
| *(inherits Minimal)* | | | |
| `configure_interrupt` | `int_pin`, `callback` | — | Attach `callback(changed_mask: int)` to the chip's INT line; `int_pin` is a hardware GPIO pin pre-configured as input (MicroPython `machine.Pin`, Linux `gpiod` line, or equivalent). On Linux, delivers interrupts via a polling thread when no hardware IRQ is available (pass `int_pin=None`). |
| `clear_interrupt` | — | int | Read PORT_IN, compare to previous read, return bitmask of pins that changed; clears the INT line. |

**Pin API additions — Full**

| Operation | MicroPython | CircuitPython | C++ | Node.js | Rust |
|-----------|-------------|---------------|-----|---------|------|
| Attach interrupt | `pin.irq(handler, Pin.IRQ_FALLING)` | *(via driver)* | `pin.attachInterrupt(handler, FALLING)` | `pin.watch(handler)` | *(via driver callback)* |
| Detach interrupt | `pin.irq(None)` | *(via driver)* | `pin.detachInterrupt()` | `pin.unwatch(handler)` | *(via driver)* |

**Additional configuration options:** none beyond what Minimal provides — the PCF8574 has no interrupt mask, polarity, or trigger-mode registers.

## GPIO Interface

This section defines the per-platform contracts for Pin objects. See `AGENTS.md` for implementation patterns.

### Python — MicroPython

Pin objects implement a compatible subset of `machine.Pin`. Constants (`IN`, `OUT`, `IRQ_FALLING`, `IRQ_RISING`) are defined as class attributes on `Pcf8574Minimal`.

Minimal interface: `init(mode)`, `value([x])`, `on()`, `off()`, `toggle()`  
Full adds: `irq(handler, trigger)` — trigger is `IRQ_FALLING` or `IRQ_RISING` (the PCF8574 INT is falling-edge; `IRQ_RISING` re-fires on the next read that shows the pin returned high).

### Python — CircuitPython

Pin objects implement `digitalio.DigitalInOut`.

Minimal interface: `direction` (read/write), `value` (read/write), `switch_to_input()`, `switch_to_output(value=False)`  
Full adds: no additional properties (`pull` and `drive_mode` are not applicable — the PCF8574 has a fixed internal pull-up and open-drain output). Both setters should raise `AttributeError` with a descriptive message.

### Python — Linux

Same interface as MicroPython (`init`, `value`, `on`, `off`, `toggle`). Interrupt delivery in Full uses a background polling thread that calls `clear_interrupt()` every 5 ms; when a change is detected the callback is invoked. If a `gpiod` line for the INT pin is passed to `configure_interrupt`, use edge detection instead of polling.

### C++

`IOExpanderPin` is a nested class in `Pcf8574Minimal`. Arduino GPIO constants are reused (`INPUT`, `OUTPUT`, `HIGH`, `LOW`).

Minimal interface:
```cpp
class IOExpanderPin {
public:
    void mode(uint8_t m);     // INPUT, OUTPUT
    void write(uint8_t v);    // HIGH (→ input mode), LOW (→ drive low)
    uint8_t read();
    void high()   { write(HIGH); }
    void low()    { write(LOW); }
    void toggle() { write(read() ^ 1); }
};
```

Full adds: `attachInterrupt(void (*handler)(uint8_t changed_mask), uint8_t mode)` and `detachInterrupt()` to `Pcf8574Full`. `mode` accepts `FALLING` only (the INT pin is active-low; any-change is resolved in software by comparing reads). On Linux, deliver via a `poll()` thread on the INT GPIO sysfs fd. On Zephyr, deliver via `gpio_add_callback()`.

### Node.js

Pin objects implement the [`onoff`](https://www.npmjs.com/package/onoff) `Gpio` interface subset.

Minimal interface: `readSync()`, `writeSync(value)`, `read(callback)`, `write(value, callback)`, `direction` property, `unexport()`  
Full adds: `watch(callback)`, `unwatch(callback)` — interrupt delivery via polling the INT GPIO using `epoll` (Linux sysfs) or via a 5 ms poll interval when no INT line is available.

### Rust

Output pins implement `embedded_hal::digital::OutputPin`; input pins implement `embedded_hal::digital::InputPin`. The driver wraps its I2C bus in `core::cell::RefCell<I2C>`.

The `pin()` factory returns an `ExPin<'_, I2C>` that implements both traits, since the PCF8574 does not distinguish direction at the register level. Callers that need a typed pin use `input_pin(n)` → `InputExPin` or `output_pin(n)` → `OutputExPin` — thin wrappers that only expose the relevant trait.

Full adds `embedded_hal::digital::StatefulOutputPin`: reading back the shadow register (not the bus) reports what was last written to the output latch.

Re-export from category `mod.rs`:
```rust
pub use pcf8574::{Pcf8574Minimal, Pcf8574Full, ExPin};
```

### JVM

The `Pin` class is a `public static class` nested inside `Pcf8574Minimal`. Interrupt delivery is always via a background polling thread; no hardware GPIO edge detection is used.

Minimal interface:
```java
chip.pin(n)          // → Pcf8574Minimal.Pin
pin.setInput()       // write 1 (release to quasi-input)
pin.setOutput()      // write 0 (drive low)
pin.setHigh()        // write 1 — alias for setInput
pin.setLow()         // write 0 — alias for setOutput
pin.read()           // → boolean (bus read, not shadow)
pin.toggle()         // invert shadow bit
```

Full adds interrupt at the driver level:
```java
chip.configureInterrupt(IntConsumer callback)  // start 5 ms polling thread
chip.clearInterrupt()                          // → int changed bitmask; clears INT
chip.stopInterrupt()                           // stop polling thread
```

In Kotlin, `configureInterrupt` accepts a `(Int) -> Unit` lambda. In Groovy, it accepts a `Closure`.  
No per-pin interrupt methods exist in the JVM interface; per-pin callbacks should be dispatched inside the driver-level callback using the changed bitmask.


## Data Conversion

```
pin n  →  port 0, bit n   (n = 0–7)

Read pin n:
    byte = read_port(0)
    value = (byte >> n) & 1

Write pin n:
    if value == 1:  shadow |= (1 << n)
    else:           shadow &= ~(1 << n)
    write_port(0, shadow)
```

The shadow register is the authoritative record of what was last written. It must be initialised to `0xFF` at construction and updated on every write operation (both `write_port` and pin-level set/clear).

### Demo flow

Four inject nodes periodically write a rotating pattern to P0–P3 (LEDs). A fifth inject polls P4–P7 (buttons) and sends the result to a debug node. A sixth inject reads the full port byte and displays it as a hex string. Shows simultaneous use of write (LED control) and read (button sense) paths.

## Examples

### Demo

Wire four LEDs (active-low, cathode to P0–P3) and four push buttons (P4–P7 to GND when pressed, floating high otherwise). The demo mirrors button inputs to LEDs: P4→P0, P5→P1, P6→P2, P7→P3. It reads the full port byte every 200 ms, extracts the button nibble (bits 4–7), inverts it (button pressed = 0 → LED on = 0), and writes the result to the output nibble (bits 0–3). The `print` shows both the raw port byte and the decoded button/LED states so the quasi-bidirectional read-back behavior is visible.

## Timing Constraints

- No startup delay required after power-on.
- INT asserts within one I²C bit period (~10 µs at 100 kHz) after an input changes.
- INT clears within one read transaction (~90 µs at 100 kHz).
- At 100 kHz, a full read-modify-write cycle (read port + write port) takes ~180 µs.

## Implementation Notes

- **No push-pull high drive.** Writing 1 enables the internal current source (~100 µA). This is sufficient to pull a bus line or read back logic levels but cannot drive a low-impedance load high. LEDs must be wired active-low (cathode to pin, anode to VCC).
- **Shadow register is mandatory.** The device has no read-back of the output latch. To set or clear individual bits, the driver must read the shadow (not the bus), modify it, and write the full byte.
- **INT is any-change on inputs only.** Pins currently driven low (shadow bit = 0) do not generate INT. Only pins with shadow bit = 1 (input mode) trigger INT when their state changes.
- **INT cleared by reading.** Any I²C read to the device address clears the INT line. The driver must save the last-read value to detect which pins changed.
- **PCF8574A address conflict.** The PCF8574A (0x38–0x3F) overlaps with common OLED display addresses. Document this in inline API docs.
- **No reset pin.** The device resets only on power cycle.
