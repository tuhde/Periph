# Chip Spec: BME280

**Manufacturer:** Bosch Sensortec  
**Datasheet:** `datasheets/environmental/bme280.pdf`  
**Category:** `environmental`  
**Transports:** I²C, SPI (3-wire and 4-wire)

## Overview

The BME280 is Bosch's combined humidity + pressure + temperature sensor. It is the BMP280 plus an integrated humidity front-end, sharing the same package, the same I²C/SPI interface, the same power modes (sleep / forced / normal), the same IIR filter, and almost the same register map. The added humidity sensor delivers ±3 %RH absolute accuracy with 1 s response time and 0.008 %RH resolution; pressure performance is identical to the BMP280 (±1 hPa absolute, 0.16 Pa noise floor with full filtering); temperature is the bridge sensor used by both compensation paths.

The BME280 is **register-compatible** with the BMP280 for pressure and temperature: the same `ctrl_meas`, `config`, `press_*`, and `temp_*` registers behave identically. The differences are: (1) the chip-ID register reads `0x60` instead of `0x58`; (2) a new `ctrl_hum` register at `0xF2` adds humidity oversampling; (3) new humidity output registers at `0xFD`/`0xFE`; (4) extra calibration coefficients at `0xA1` and `0xE1`–`0xE7`; (5) `t_sb` codes `110` and `111` mean **10 ms / 20 ms** on the BME280 (vs. 2000 ms / 4000 ms on the BMP280) — a register-value incompatibility, not just an extension. The compensation algorithm for humidity uses `t_fine` from the temperature path, like the pressure compensation does. See `specs/pressure/bmp280.md` for the heritage.

## Transport Configuration

### I²C

- **Address:** `0x76` (SDO = GND) or `0x77` (SDO = V_DDIO)
- **Max clock:** 3.4 MHz (high-speed mode)
- **Pull-ups:** required on SDA and SCL; CSB must be tied to V_DDIO to select I²C mode

`0x77` collides with the BMP180, BMP280, and BMP388 on the same bus — use `0x76` if any of those share a bus.

### SPI

- **Mode:** CPOL = CPHA = 0 (mode 00) or CPOL = CPHA = 1 (mode 11). Auto-selected by SCK level at the falling edge of CSB.
- **Max clock:** 10 MHz
- **Bit order:** MSB first
- **CS active:** low (CSB pin)
- **Wire mode:** 4-wire by default; set `config.spi3w_en = 1` for 3-wire (SDI bidirectional, SDO hi-Z)
- **Register-address protocol:** SPI uses 7-bit register addresses with bit 7 as the R/W flag. **Read:** send `reg_addr | 0x80` (RW = 1), then read N bytes. **Write:** send `reg_addr & 0x7F` (RW = 0), then send data.

All BME280 register addresses have bit 7 set, so a chip-side read protocol of "send `reg` to start a read" produces identical bytes on both buses; writes differ (I²C uses the full address, SPI uses `reg & 0x7F`).

**Initial implementation scope:** as with the BMP280, the project currently has SPI transport infrastructure only on Arduino C++. The first implementation pass should target **I²C across all platforms**; SPI is documented in this spec for completeness but is deferred until the project gains a register-aware SPI transport for the other languages. See `specs/pressure/bmp280.md` Implementation Notes for the same caveat at length.

## Register Map

| Address    | Name             | R/W | Reset    | Description |
|------------|------------------|-----|----------|-------------|
| `0x88`–`0xA1` | Calibration NVM (block 1) | R | factory | `dig_T1`–`dig_T3`, `dig_P1`–`dig_P9`, `dig_H1` (26 bytes) |
| `0xD0`     | `id`             | R   | `0x60`   | Fixed chip-ID (BMP280 was `0x58`) |
| `0xE0`     | `reset`          | W   | —        | Write `0xB6` to perform a power-on reset |
| `0xE1`–`0xE7` | Calibration NVM (block 2) | R | factory | `dig_H2`–`dig_H6` (7 bytes, with `dig_H4`/`dig_H5` packed) |
| `0xF2`     | `ctrl_hum`       | R/W | `0x00`   | `osrs_h<2:0>` — **must be written before `ctrl_meas` to take effect** |
| `0xF3`     | `status`         | R   | `0x00`   | Bit 3 = `measuring`, bit 0 = `im_update` |
| `0xF4`     | `ctrl_meas`      | R/W | `0x00`   | `osrs_t<7:5>`, `osrs_p<4:2>`, `mode<1:0>` |
| `0xF5`     | `config`         | R/W | `0x00`   | `t_sb<7:5>`, `filter<4:2>`, `spi3w_en<0>` |
| `0xF7`–`0xF9` | `press_msb` / `_lsb` / `_xlsb` | R | `0x80…00` | 20-bit pressure ADC |
| `0xFA`–`0xFC` | `temp_msb` / `_lsb` / `_xlsb`  | R | `0x80…00` | 20-bit temperature ADC |
| `0xFD`–`0xFE` | `hum_msb` / `_lsb`             | R | `0x80…00` | 16-bit humidity ADC |

### Calibration NVM

Block 1 (`0x88`–`0xA1`, 26 bytes, **little-endian** for the 16-bit fields):

| LSB / MSB         | Name    | Type   |
|-------------------|---------|--------|
| `0x88` / `0x89`   | dig_T1  | uint16 |
| `0x8A` / `0x8B`   | dig_T2  | int16  |
| `0x8C` / `0x8D`   | dig_T3  | int16  |
| `0x8E` / `0x8F`   | dig_P1  | uint16 |
| `0x90` / `0x91`   | dig_P2  | int16  |
| `0x92` / `0x93`   | dig_P3  | int16  |
| `0x94` / `0x95`   | dig_P4  | int16  |
| `0x96` / `0x97`   | dig_P5  | int16  |
| `0x98` / `0x99`   | dig_P6  | int16  |
| `0x9A` / `0x9B`   | dig_P7  | int16  |
| `0x9C` / `0x9D`   | dig_P8  | int16  |
| `0x9E` / `0x9F`   | dig_P9  | int16  |
| `0xA0`            | reserved | —     |
| `0xA1`            | dig_H1  | uint8  |

(`0xA0` is not used; just skip that byte.)

Block 2 (`0xE1`–`0xE7`, 7 bytes — note the unusual packing of `dig_H4` / `dig_H5`):

| Address(es)            | Name    | Type     | Encoding |
|------------------------|---------|----------|----------|
| `0xE1` / `0xE2`        | dig_H2  | int16    | LE pair |
| `0xE3`                 | dig_H3  | uint8    | single byte |
| `0xE4`, `0xE5[3:0]`    | dig_H4  | int16    | `(b[0xE4] << 4) | (b[0xE5] & 0x0F)`, **then sign-extend from 12 bits** |
| `0xE5[7:4]`, `0xE6`    | dig_H5  | int16    | `(b[0xE6] << 4) | (b[0xE5] >> 4)`, **then sign-extend from 12 bits** |
| `0xE7`                 | dig_H6  | int8     | single signed byte |

`dig_H4` and `dig_H5` share register `0xE5` (lower nibble for `dig_H4`, upper nibble for `dig_H5`), and both are 12-bit two's-complement values stored in 16-bit slots. Sign-extend by checking bit 11 of the assembled value: if set, OR with `0xF000`.

Burst-read both blocks separately: 26 bytes from `0x88`, then 7 bytes from `0xE1`. (Alternatively, one 33-byte burst from `0x88`–`0xA8` and a separate 7-byte burst — but reading the reserved gap is wasteful.)

### `ctrl_hum` (`0xF2`)

| Bits | Name     | Description |
|------|----------|-------------|
| 7:3  | —        | Reserved (write 0) |
| 2:0  | `osrs_h` | Humidity oversampling: `000`=skipped, `001`=×1, `010`=×2, `011`=×4, `100`=×8, `101`+=×16 |

**Critical:** changes to `ctrl_hum` only take effect **after a write to `ctrl_meas`**. The chip ignores intermediate `ctrl_hum` writes between `ctrl_meas` writes. The driver's `configure()` / `set_oversampling()` sequence must always be: write `ctrl_hum` first, then write `ctrl_meas`.

### `ctrl_meas` (`0xF4`) — same as BMP280

| Bits | Name    | Description |
|------|---------|-------------|
| 7:5  | `osrs_t` | Temperature oversampling: `000`=skipped, `001`=×1, …, `101`+=×16 |
| 4:2  | `osrs_p` | Pressure oversampling: same encoding |
| 1:0  | `mode`   | `00`=Sleep, `01`/`10`=Forced, `11`=Normal |

### `config` (`0xF5`)

| Bits | Name      | Description |
|------|-----------|-------------|
| 7:5  | `t_sb`    | Standby time in normal mode (see table below — **differs from BMP280 at codes 110/111**) |
| 4:2  | `filter`  | IIR filter: `000`=off, `001`=2, `010`=4, `011`=8, `100`+=16 |
| 1    | —         | Reserved |
| 0    | `spi3w_en` | `1` enables 3-wire SPI |

**`t_sb` (BME280 values):**

| Value | t_standby |
|-------|-----------|
| 000   | 0.5 ms |
| 001   | 62.5 ms |
| 010   | 125 ms |
| 011   | 250 ms |
| 100   | 500 ms |
| 101   | 1000 ms |
| 110   | **10 ms** |
| 111   | **20 ms** |

The bottom two values are **the BME280's only register-value incompatibility with the BMP280**. A BMP280 driver that maps `t_sb = 7` to "4000 ms" will produce ~200× faster sampling on a BME280. Drivers that share code between the two chips must branch here.

### `status` (`0xF3`) — same as BMP280

| Bit | Name        | Description |
|-----|-------------|-------------|
| 3   | `measuring` | `1` while a conversion is running |
| 0   | `im_update` | `1` while NVM is being copied to image registers |

### Output registers

- Pressure (`0xF7`–`0xF9`): same layout as BMP280, 20-bit unsigned left-aligned. `adc_P = (raw[0] << 12) | (raw[1] << 4) | (raw[2] >> 4)`.
- Temperature (`0xFA`–`0xFC`): same layout, 20-bit unsigned. `adc_T = (raw[0] << 12) | (raw[1] << 4) | (raw[2] >> 4)`.
- Humidity (`0xFD`–`0xFE`): 16-bit unsigned, big-endian. `adc_H = (raw[0] << 8) | raw[1]`.

A burst read from `0xF7` to `0xFE` returns all 8 bytes in one transaction, which is the recommended way to read all three measurements without tearing.

## Initialization Sequence

### Minimal

1. (Optional) Read `id` (`0xD0`) and confirm `0x60`.
2. Burst-read 26 bytes from `0x88`–`0xA1` (skipping `0xA0`) and 7 bytes from `0xE1`–`0xE7`; unpack into 18 trimming coefficients.
3. Write `ctrl_hum` (`0xF2`) with `osrs_h = ×1`.
4. Write `ctrl_meas` (`0xF4`) with `osrs_t = ×1`, `osrs_p = ×1`, `mode = 00` (sleep) — **must follow `ctrl_hum`** for that write to latch.
5. (Recommended) Write `config` (`0xF5`) with `filter = off`, `t_sb` = don't care, `spi3w_en = 0`.

The driver triggers each measurement explicitly in **forced mode** by writing `ctrl_meas` with `mode = 01`.

### Full

Same as Minimal, but the user may pass `osrs_t`, `osrs_p`, `osrs_h`, `filter`, `t_sb`, and `mode` to the constructor or change them later. Any change to `osrs_h` must be followed by a `ctrl_meas` write; the driver's setters handle this.

## Implementation Stages

### Minimal

Goal: read calibrated temperature (°C), pressure (hPa), and humidity (%RH) with no configuration beyond the transport.

Constructor parameters: `transport`. The constructor reads and stores all 18 trimming coefficients.

**Baked-in defaults:**
- Power mode: forced (each measurement call triggers a single-shot conversion)
- `osrs_t = ×1`, `osrs_p = ×1`, `osrs_h = ×1` (Ultra Low Power, weather-monitoring preset)
- IIR filter off
- `spi3w_en = 0`

| Operation       | Parameters | Returns | Notes |
|-----------------|------------|---------|-------|
| `temperature()` | —          | float   | °C; caches `t_fine` for use in pressure & humidity compensation |
| `pressure()`    | —          | float   | hPa; triggers a forced measurement and re-reads UT/UP, refreshing `t_fine` |
| `humidity()`    | —          | float   | %RH; triggers a forced measurement and re-reads UT/UH, refreshing `t_fine` |

A single forced-mode trigger samples all three ADCs (when none of `osrs_*` is skipped) in one cycle. The driver triggers once per call but reads all 8 output bytes (`0xF7`–`0xFE`) in one burst regardless of which value the caller asked for, so the cost is the same per call. Per the datasheet's "weather monitoring" preset, the wall-clock cost at default oversampling is ~9 ms.

### Full

Extends Minimal. Adds power modes, oversampling for all three channels, IIR filter, standby time, and altitude helpers.

| Operation                  | Parameters                                                          | Returns | Notes |
|----------------------------|---------------------------------------------------------------------|---------|-------|
| *(inherits Minimal)*       | | | |
| `configure()`              | `osrs_t`, `osrs_p`, `osrs_h`, `mode`, `filter`, `t_sb`              | —       | Writes `ctrl_hum`, `ctrl_meas`, and `config` in the correct order |
| `set_oversampling()`       | `osrs_t`, `osrs_p`, `osrs_h`                                        | —       | Update oversampling (writes `ctrl_hum` then `ctrl_meas`) |
| `set_mode()`               | `mode`                                                              | —       | Update `mode` bits of `ctrl_meas` |
| `set_filter()`             | `coeff`                                                             | —       | Update `filter` bits of `config` |
| `set_standby()`            | `t_sb`                                                              | —       | Update `t_sb` bits of `config` |
| `status()`                 | —                                                                   | int     | Read `0xF3`; constants `STATUS_MEASURING` (`0x08`), `STATUS_IM_UPDATE` (`0x01`) |
| `altitude()`               | `sea_level_hpa=1013.25`                                             | float   | Meters above sea level |
| `sea_level_pressure()`     | `altitude_m`                                                        | float   | Sea-level pressure (hPa) implied by current pressure and a known altitude |
| `dew_point()`              | —                                                                   | float   | °C; computed from temperature and humidity using Magnus-Tetens approximation |
| `chip_id()`                | —                                                                   | int     | Read `0xD0`; expect `0x60` |
| `reset()`                  | —                                                                   | —       | Soft reset (write `0xB6` to `0xE0`); re-reads calibration and re-applies user configuration |

**Constants exposed alongside the Full class** (mirror BMP280 plus humidity):

Oversampling: `OSRS_SKIP = 0`, `OSRS_X1 = 1`, `OSRS_X2 = 2`, `OSRS_X4 = 3`, `OSRS_X8 = 4`, `OSRS_X16 = 5`  
Mode: `MODE_SLEEP = 0`, `MODE_FORCED = 1`, `MODE_NORMAL = 3`  
Filter: `FILTER_OFF = 0`, `FILTER_2 = 1`, `FILTER_4 = 2`, `FILTER_8 = 3`, `FILTER_16 = 4`  
Standby: `T_SB_0_5_MS = 0`, `T_SB_62_5_MS = 1`, `T_SB_125_MS = 2`, `T_SB_250_MS = 3`, `T_SB_500_MS = 4`, `T_SB_1000_MS = 5`, **`T_SB_10_MS = 6`**, **`T_SB_20_MS = 7`**

(Note the difference from BMP280's standby constants — codes 6 and 7 mean different times on the BME280.)

In **normal mode**, calls to `temperature()` / `pressure()` / `humidity()` should not re-trigger; just read the most recent shadow registers. The driver tracks the active mode.

## Data Conversion

Pressure and temperature compensation are **identical to the BMP280** (datasheet section 4.2.3). The BME280 adds a humidity compensation that uses `t_fine`. Use the 64-bit integer path; a 32-bit pressure fallback is in datasheet Appendix 8.2.

```
# After burst-reading registers 0xF7..0xFE (8 bytes):
adc_P = (raw[0] << 12) | (raw[1] << 4) | (raw[2] >> 4)   # 20-bit unsigned
adc_T = (raw[3] << 12) | (raw[4] << 4) | (raw[5] >> 4)   # 20-bit unsigned
adc_H = (raw[6] << 8)  |  raw[7]                          # 16-bit unsigned

# Temperature (returns °C; updates t_fine, signed 32-bit):
var1   = (((adc_T >> 3) - (dig_T1 << 1)) * dig_T2) >> 11
var2   = (((((adc_T >> 4) - dig_T1) * ((adc_T >> 4) - dig_T1)) >> 12) * dig_T3) >> 14
t_fine = var1 + var2
T      = ((t_fine * 5 + 128) >> 8) / 100.0               # °C

# Pressure (64-bit path; identical to BMP280; returns Pa as Q24.8 → divide by 256):
# (see specs/pressure/bmp280.md for the full algorithm)

# Humidity (32-bit integer; returns %RH as Q22.10 → divide by 1024):
v = t_fine - 76800
v = (((((adc_H << 14) - (dig_H4 << 20) - (dig_H5 * v)) + 16384) >> 15) *
     (((((((v * dig_H6) >> 10) * (((v * dig_H3) >> 11) + 32768)) >> 10) + 2097152) *
        dig_H2 + 8192) >> 14))
v = v - (((((v >> 15) * (v >> 15)) >> 7) * dig_H1) >> 4)
v = 0 if v < 0 else v
v = 419430400 if v > 419430400 else v
H = (v >> 12) / 1024.0                                    # %RH (clamped to 0..100)
```

The humidity compensation requires `t_fine` from a recent temperature reading; the driver always reads all three ADCs in one burst, so `t_fine` is always fresh when humidity is computed.

### Validation

Use the BMP280's worked example to validate the pressure and temperature paths (see `specs/pressure/bmp280.md`); the BME280 produces identical values for the same trimming coefficients and ADC values. There is no published worked example for the humidity compensation in the datasheet, so unit tests should cross-check against the Bosch reference C source code or an independent implementation (e.g., Adafruit_BME280) at known calibration values and `adc_H`.

### Altitude / sea-level / dew point

Altitude and sea-level conversions are the same international barometric formulas as the BMP180/BMP280:

```
altitude_m    = 44330 * (1 - (p_hpa / sea_level_hpa) ** (1 / 5.255))
sea_level_hpa = p_hpa / (1 - altitude_m / 44330) ** 5.255
```

Dew point uses the Magnus-Tetens approximation (accurate to ±0.4 °C in the 0–60 °C / 1–100 %RH range):

```
a = 17.27
b = 237.7
α = (a * T) / (b + T) + ln(RH / 100)
dew_point_C = (b * α) / (a - α)
```

## Examples

### Demo

A small indoor environmental logger. The demo polls all three sensors once per minute for 10 minutes (600 s total) using the datasheet's "weather monitoring" preset (forced mode, ×1/×1/×1 oversampling, filter off — minimum power). Each tick prints `T / RH / P / dew_point / altitude` on one line. Half-way through, the demo prompts the user to breathe gently on the sensor for 3 seconds; the user sees humidity climb from ~40 %RH to ~80 %RH and the dew point spike, while pressure stays flat and temperature rises only slightly. After the run, prints min / max / mean of all five values to demonstrate that the BME280 is one chip doing the work of three.

## Timing Constraints

- Power-on / soft-reset start-up time before first communication: 2 ms
- Forced-mode measurement times depend on the sum of enabled oversampling settings. Per datasheet section 9.1:

  | Preset                  | osrs_t | osrs_p | osrs_h | typ      | max      |
  |-------------------------|--------|--------|--------|----------|----------|
  | Weather monitoring      | ×1     | ×1     | ×1     | 8 ms     | 9.3 ms   |
  | Humidity sensing        | skip   | ×1     | ×1     | 4 ms     | 4.6 ms   |
  | Indoor navigation       | ×2     | ×16    | ×1     | 38.5 ms  | 44.4 ms  |
  | Gaming                  | ×1     | ×4     | skip   | 11.5 ms  | 13.3 ms  |

  Drivers should sleep for the **max** time after triggering forced mode, then read.
- Normal-mode standby times: see the `t_sb` table above. The 0.5 ms entry plus minimum measurement time gives the maximum sustained ODR (~157 Hz at lowest oversampling).
- IIR filter settling: `samples_to_75% × t_meas`. For `filter = 16` at indoor-navigation settings (~25 Hz ODR), settling is ~0.9 s.

## Implementation Notes

- **Heritage:** the BME280 driver is a sibling, not a subclass, of the BMP280 driver. They share register addresses for pressure and temperature and share the temperature/pressure compensation algorithm — but the calibration block, `ctrl_hum` register, humidity output, humidity compensation, chip-ID, and `t_sb` codes 6–7 differ. Implementations should not try to share a base class with BMP280; copy the relevant pieces and add humidity. (For language-specific layouts, this might be `BME280Minimal` extending `BME280Full` directly with no parent — same shape as BMP280.)
- **`ctrl_hum` write order matters.** Always write `ctrl_hum` *before* `ctrl_meas`. The chip ignores `ctrl_hum` writes in isolation; the value latches when `ctrl_meas` is next written. This is the most common BME280 bug — humidity reads come back as the same constant value forever because oversampling never gets applied.
- **`dig_H4` and `dig_H5` share a register.** `dig_H4 = (b[0xE4] << 4) | (b[0xE5] & 0x0F)`, `dig_H5 = (b[0xE6] << 4) | (b[0xE5] >> 4)`. Both are 12-bit signed values; sign-extend to 16-bit by checking bit 11. A bit-shift error here silently corrupts the humidity reading by 10–20 %RH at room conditions.
- **64-bit integer math** is recommended for the pressure compensation (same as BMP280) and **32-bit signed math** for the humidity compensation (datasheet rev 1.1 reference C). On platforms where 64-bit ints are problematic (8-bit MCUs), use the 32-bit pressure fallback in Appendix 8.2.
- **Burst-read all 8 output bytes** from `0xF7` to `0xFE` in one transaction. Reading them as separate transactions can interleave bytes from two measurement cycles in normal mode (datasheet section 4.1, data-register shadowing).
- **`t_fine` is shared** by all three compensation paths. The driver caches it after every read, then reuses it for pressure and humidity compensation (which never re-derive it).
- **Single forced-mode trigger samples all three ADCs** when no oversampling is set to skip. The driver reads all 8 output bytes regardless of which method the caller invoked, so calling `temperature()`, `pressure()`, and `humidity()` in sequence costs three forced-mode triggers (3× the wait time). For best efficiency on tight loops, expose a `read_all() -> (T, p, RH)` method as part of Full — *but only if the API stays simple*; the three single-value methods are the canonical interface.
- **`0x77` collides** with the BMP180, BMP280, BMP388, and several non-Bosch sensors. The BME280's chip-ID `0x60` distinguishes it from those at runtime.
- **In normal mode**, the chip cycles continuously. The driver must track the active mode and not re-trigger reads in normal mode — just read shadow registers. In sleep / forced mode, every read triggers a fresh forced-mode conversion.
- **Soft reset** (`0xB6` → `0xE0`) takes the chip back to power-on state. Trimming NVM survives but the driver must re-read it (defensive) and re-apply user-configured `ctrl_hum` / `ctrl_meas` / `config` values.
- **`t_sb` codes 110/111 differ from BMP280.** A driver that copy-pastes BMP280's standby table will produce ~200× faster sampling at those codes. Use the BME280's table verbatim (`10 ms` / `20 ms`).
- **SPI scope:** initial implementation is I²C-only. The chip driver should write read addresses with bit 7 set (which is the natural address) so that a future register-aware SPI transport will "just work" on reads. Writes need the bus type known — defer until the project has SPI infrastructure across more languages.
