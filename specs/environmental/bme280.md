# Chip Spec: BME280

**Manufacturer:** Bosch Sensortec  
**Datasheet:** `datasheets/environmental/bme280.pdf`  
**Category:** `environmental`  
**Transports:** IÂ²C, SPI (3-wire and 4-wire)

## Overview

The BME280 is Bosch's combined humidity + pressure + temperature sensor. It is the BMP280 plus an integrated humidity front-end, sharing the same package, the same IÂ²C/SPI interface, the same power modes (sleep / forced / normal), the same IIR filter, and almost the same register map. The added humidity sensor delivers Â±3 %RH absolute accuracy with 1 s response time and 0.008 %RH resolution; pressure performance is identical to the BMP280 (Â±1 hPa absolute, 0.16 Pa noise floor with full filtering); temperature is the bridge sensor used by both compensation paths.

The BME280 is **register-compatible** with the BMP280 for pressure and temperature: the same `ctrl_meas`, `config`, `press_*`, and `temp_*` registers behave identically. The differences are: (1) the chip-ID register reads `0x60` instead of `0x58`; (2) a new `ctrl_hum` register at `0xF2` adds humidity oversampling; (3) new humidity output registers at `0xFD`/`0xFE`; (4) extra calibration coefficients at `0xA1` and `0xE1`â€“`0xE7`; (5) `t_sb` codes `110` and `111` mean **10 ms / 20 ms** on the BME280 (vs. 2000 ms / 4000 ms on the BMP280) â€” a register-value incompatibility, not just an extension. The compensation algorithm for humidity uses `t_fine` from the temperature path, like the pressure compensation does. See `specs/pressure/bmp280.md` for the heritage.

## Transport Configuration

### IÂ²C

- **Address:** `0x76` (SDO = GND) or `0x77` (SDO = V_DDIO)
- **Max clock:** 3.4 MHz (high-speed mode)
- **Pull-ups:** required on SDA and SCL; CSB must be tied to V_DDIO to select IÂ²C mode

`0x77` collides with the BMP180, BMP280, and BMP388 on the same bus â€” use `0x76` if any of those share a bus.

### SPI

- **Mode:** CPOL = CPHA = 0 (mode 00) or CPOL = CPHA = 1 (mode 11). Auto-selected by SCK level at the falling edge of CSB.
- **Max clock:** 10 MHz
- **Bit order:** MSB first
- **CS active:** low (CSB pin)
- **Wire mode:** 4-wire by default; set `config.spi3w_en = 1` for 3-wire (SDI bidirectional, SDO hi-Z)
- **Register-address protocol:** SPI uses 7-bit register addresses with bit 7 as the R/W flag. **Read:** send `reg_addr | 0x80` (RW = 1), then read N bytes. **Write:** send `reg_addr & 0x7F` (RW = 0), then send data.

All BME280 register addresses have bit 7 set, so a chip-side read protocol of "send `reg` to start a read" produces identical bytes on both buses; writes differ (IÂ²C uses the full address, SPI uses `reg & 0x7F`).

**SPI is now supported across all platforms** â€” `SPITransport` variants exist for MicroPython, CircuitPython, Linux (spidev), Arduino, Zephyr, and Rust (embedded-hal `SpiDevice`). See `specs/transport_spi.md`. Chip drivers may implement SPI from the initial pass; see the bit-7 write-address caveat in `specs/pressure/bmp280.md` Implementation Notes (same protocol applies here).

| Address(es)            | Name    | Type     | Encoding |
|------------------------|---------|----------|----------|
| `0xE1` / `0xE2`        | dig_H2  | int16    | LE pair |
| `0xE3`                 | dig_H3  | uint8    | single byte |

`dig_H4` and `dig_H5` share register `0xE5` (lower nibble for `dig_H4`, upper nibble for `dig_H5`), and both are 12-bit two's-complement values stored in 16-bit slots. Sign-extend by checking bit 11 of the assembled value: if set, OR with `0xF000`.

Burst-read both blocks separately: 26 bytes from `0x88`, then 7 bytes from `0xE1`. (Alternatively, one 33-byte burst from `0x88`â€“`0xA8` and a separate 7-byte burst â€” but reading the reserved gap is wasteful.)

### Output registers

- Pressure (`0xF7`â€“`0xF9`): same layout as BMP280, 20-bit unsigned left-aligned. `adc_P = (raw[0] << 12) | (raw[1] << 4) | (raw[2] >> 4)`.
- Temperature (`0xFA`â€“`0xFC`): same layout, 20-bit unsigned. `adc_T = (raw[0] << 12) | (raw[1] << 4) | (raw[2] >> 4)`.
- Humidity (`0xFD`â€“`0xFE`): 16-bit unsigned, big-endian. `adc_H = (raw[0] << 8) | raw[1]`.

A burst read from `0xF7` to `0xFE` returns all 8 bytes in one transaction, which is the recommended way to read all three measurements without tearing.

## Initialization Sequence

### Minimal

1. (Optional) Read `id` (`0xD0`) and confirm `0x60`.
2. Burst-read 26 bytes from `0x88`â€“`0xA1` (skipping `0xA0`) and 7 bytes from `0xE1`â€“`0xE7`; unpack into 18 trimming coefficients.
3. Write `ctrl_hum` (`0xF2`) with `osrs_h = Ã—1`.
4. Write `ctrl_meas` (`0xF4`) with `osrs_t = Ã—1`, `osrs_p = Ã—1`, `mode = 00` (sleep) â€” **must follow `ctrl_hum`** for that write to latch.
5. (Recommended) Write `config` (`0xF5`) with `filter = off`, `t_sb` = don't care, `spi3w_en = 0`.

The driver triggers each measurement explicitly in **forced mode** by writing `ctrl_meas` with `mode = 01`.

### Full

Same as Minimal, but the user may pass `osrs_t`, `osrs_p`, `osrs_h`, `filter`, `t_sb`, and `mode` to the constructor or change them later. Any change to `osrs_h` must be followed by a `ctrl_meas` write; the driver's setters handle this.

## Data Conversion

Pressure and temperature compensation are **identical to the BMP280** (datasheet section 4.2.3). The BME280 adds a humidity compensation that uses `t_fine`. Use the 64-bit integer path; a 32-bit pressure fallback is in datasheet Appendix 8.2.

```
# After burst-reading registers 0xF7..0xFE (8 bytes):
adc_P = (raw[0] << 12) | (raw[1] << 4) | (raw[2] >> 4)   # 20-bit unsigned
adc_T = (raw[3] << 12) | (raw[4] << 4) | (raw[5] >> 4)   # 20-bit unsigned
adc_H = (raw[6] << 8)  |  raw[7]                          # 16-bit unsigned

# Temperature (returns Â°C; updates t_fine, signed 32-bit):
var1   = (((adc_T >> 3) - (dig_T1 << 1)) * dig_T2) >> 11
var2   = (((((adc_T >> 4) - dig_T1) * ((adc_T >> 4) - dig_T1)) >> 12) * dig_T3) >> 14
t_fine = var1 + var2
T      = ((t_fine * 5 + 128) >> 8) / 100.0               # Â°C

# Pressure (64-bit path; identical to BMP280; returns Pa as Q24.8 â†’ divide by 256):
# (see specs/pressure/bmp280.md for the full algorithm)

# Humidity (32-bit integer; returns %RH as Q22.10 â†’ divide by 1024):
v = t_fine - 76800
v = (((((adc_H << 14) - (dig_H4 << 20) - (dig_H5 * v)) + 16384) >> 15) *
     (((((((v * dig_H6) >> 10) * (((v * dig_H3) >> 11) + 32768)) >> 10) + 2097152) *
        dig_H2 + 8192) >> 14))
v = v - (((((v >> 15) * (v >> 15)) >> 7) * dig_H1) >> 4)
v = 0 if v < 0 else v
v = 419430400 if v > 419430400 else v
H = (v >> 12) / 1024.0                                    # %RH (clamped to 0..100)
```

The humidity compensation requires `t_fine` from a recent temperature reading; the driver always reads all three ADCs in one burst, so `t_fine` is always fresh when humidity is computed.

## Implementation Notes

- **Heritage:** the BME280 driver is a sibling, not a subclass, of the BMP280 driver. They share register addresses for pressure and temperature and share the temperature/pressure compensation algorithm â€” but the calibration block, `ctrl_hum` register, humidity output, humidity compensation, chip-ID, and `t_sb` codes 6â€“7 differ. Implementations should not try to share a base class with BMP280; copy the relevant pieces and add humidity. (For language-specific layouts, this might be `BME280Minimal` extending `BME280Full` directly with no parent â€” same shape as BMP280.)
- **`ctrl_hum` write order matters.** Always write `ctrl_hum` *before* `ctrl_meas`. The chip ignores `ctrl_hum` writes in isolation; the value latches when `ctrl_meas` is next written. This is the most common BME280 bug â€” humidity reads come back as the same constant value forever because oversampling never gets applied.
- **`dig_H4` and `dig_H5` share a register.** `dig_H4 = (b[0xE4] << 4) | (b[0xE5] & 0x0F)`, `dig_H5 = (b[0xE6] << 4) | (b[0xE5] >> 4)`. Both are 12-bit signed values; sign-extend to 16-bit by checking bit 11. A bit-shift error here silently corrupts the humidity reading by 10â€“20 %RH at room conditions.
- **64-bit integer math** is recommended for the pressure compensation (same as BMP280) and **32-bit signed math** for the humidity compensation (datasheet rev 1.1 reference C). On platforms where 64-bit ints are problematic (8-bit MCUs), use the 32-bit pressure fallback in Appendix 8.2.
- **Burst-read all 8 output bytes** from `0xF7` to `0xFE` in one transaction. Reading them as separate transactions can interleave bytes from two measurement cycles in normal mode (datasheet section 4.1, data-register shadowing).
- **`t_fine` is shared** by all three compensation paths. The driver caches it after every read, then reuses it for pressure and humidity compensation (which never re-derive it).
- **Single forced-mode trigger samples all three ADCs** when no oversampling is set to skip. The driver reads all 8 output bytes regardless of which method the caller invoked, so calling `temperature()`, `pressure()`, and `humidity()` in sequence costs three forced-mode triggers (3Ã— the wait time). For best efficiency on tight loops, expose a `read_all() -> (T, p, RH)` method as part of Full â€” *but only if the API stays simple*; the three single-value methods are the canonical interface.
- **`0x77` collides** with the BMP180, BMP280, BMP388, and several non-Bosch sensors. The BME280's chip-ID `0x60` distinguishes it from those at runtime.
- **In normal mode**, the chip cycles continuously. The driver must track the active mode and not re-trigger reads in normal mode â€” just read shadow registers. In sleep / forced mode, every read triggers a fresh forced-mode conversion.
- **Soft reset** (`0xB6` â†’ `0xE0`) takes the chip back to power-on state. Trimming NVM survives but the driver must re-read it (defensive) and re-apply user-configured `ctrl_hum` / `ctrl_meas` / `config` values.
- **`t_sb` codes 110/111 differ from BMP280.** A driver that copy-pastes BMP280's standby table will produce ~200Ã— faster sampling at those codes. Use the BME280's table verbatim (`10 ms` / `20 ms`).
- **SPI write-address caveat:** all BME280 register addresses have bit 7 set. Reads work unchanged on SPI (natural address = read address). Writes on SPI must clear bit 7 (`reg & 0x7F`). Drivers that support both buses should accept a `bus_type` constructor argument and apply the mask only on SPI paths.
