# Chip Spec: BME680

**Manufacturer:** Bosch Sensortec  
**Datasheet:** `datasheets/environmental/bme680.pdf`  
**Category:** `environmental`  
**Transports:** I²C, SPI (3-wire and 4-wire, **paged**)

## Overview

The BME680 is Bosch's 4-in-1 environmental sensor: temperature, pressure, humidity, and a metal-oxide (MOX) gas sensor that measures total VOC content of indoor air. It builds on the BME280's TPH front-end and adds a heated gas-sensitive layer with a 10-slot heater profile and an internal control loop that targets a user-specified heater temperature in Ohms. Pressure / temperature / humidity performance is comparable to the BME280 (±1 hPa, ±1 °C, ±3 %RH); the gas sensor outputs a raw resistance whose interpretation as an air-quality index requires Bosch's BSEC software (closed-source binary, **not** part of this driver).

The BME680 is not register-compatible with the BME280 / BMP280, despite sharing some superficial structure. Differences worth flagging up front:

- **Only Sleep (`00`) and Forced (`01`) power modes** — no normal mode. Every reading is a single-shot triggered measurement.
- **Calibration block is split** across two non-contiguous register ranges (`0x8A`–`0xA0` and `0xE1`–`0xEE`) plus three single-byte values at `0x00`, `0x02`, `0x04`. The 16-bit fields are little-endian, like the BME280, but **the parameter names and addresses are different** (`par_T1` / `par_T2` / `par_T3` instead of BME280's `dig_T1`–`dig_T3`, and at different offsets — `par_T1` lives at `0xE9`/`0xEA`, not `0x88`).
- **Compensation formulas are new.** Pressure uses 32-bit integer math (no 64-bit needed); humidity has its own formula; gas resistance uses 64-bit integer math with two 16-entry lookup tables (`const_array1_int`, `const_array2_int`) baked into the driver.
- **Chip-ID is `0x61`**, not `0x60` (BME280) or `0x58` (BMP280).
- **SPI memory map is paged.** Page 0 (`0x80`–`0xFF`) and Page 1 (`0x00`–`0x7F`) share the 7-bit SPI address space; the active page is selected by bit 4 of register `0x73` (`spi_mem_page`). I²C has a flat address space and no paging. This is a substantial complication versus BMP280 / BME280 SPI.

The "indoor air quality (IAQ) index" output that the BME680 is famous for is computed by Bosch's BSEC software, not by the chip. This driver exposes the **raw gas-sensor resistance in Ohms**; computing IAQ from that resistance requires either BSEC (closed-source) or a custom heuristic. Not in scope for this driver.

## Transport Configuration

### I²C

- **Address:** `0x76` (SDO = GND) or `0x77` (SDO = V_DDIO) — same as BME280 / BMP280
- **Max clock:** 3.4 MHz (high-speed mode)
- **Pull-ups:** required on SDA and SCL; CSB must be tied to V_DDIO to select I²C mode

`0x77` collides with the BMP180 / BMP280 / BME280 / BMP388 — use `0x76` if any share a bus.

### SPI

- **Mode:** CPOL = CPHA = 0 (mode 00) or CPOL = CPHA = 1 (mode 11). Auto-selected by SCK level at the falling edge of CSB.
- **Max clock:** 10 MHz
- **Bit order:** MSB first
- **CS active:** low (CSB pin)
- **Wire mode:** 4-wire by default; set `config.spi3w_en = 1` for 3-wire (SDI bidirectional, SDO hi-Z)
- **Register-address protocol:** SPI uses 7-bit addresses with bit 7 as R/W. **Read:** send `reg_addr | 0x80`. **Write:** send `reg_addr & 0x7F`.
- **Paged memory map:** the SPI 7-bit address space holds **two pages** of 128 bytes each. The active page is selected by `spi_mem_page` (register `0x73` bit 4). Page 0 mirrors I²C addresses `0x80`–`0xFF`; Page 1 mirrors I²C addresses `0x00`–`0x7F`. After power-on, Page 0 is active.

  | Page | I²C address range it mirrors | Contains |
  |------|------------------------------|----------|
  | 0    | `0x80`–`0xFF`                | `id` (`0xD0` → SPI `0x50`), `reset` (`0xE0` → SPI `0x60`), the higher calibration block, the page-select bit at SPI `0x73` |
  | 1    | `0x00`–`0x7F`                | All data registers, all control registers, the heater-profile registers, the lower calibration values, and the page-select bit at SPI `0x73` |

  The page-select register (`0x73`) is accessible from **both** pages, so a driver in page 0 can switch to page 1 without an intermediate read. To read data registers in SPI mode, set page 1; to read `id` or to issue `reset`, set page 0. The driver must track the active page and switch lazily before each access.

**Initial implementation scope:** as with BMP280/BME280, the project currently has SPI transport infrastructure only on Arduino C++. Initial implementation is **I²C-only across all platforms**. SPI is documented in this spec for completeness; adding it later requires both a register-aware SPI transport (project-wide) and a page-aware BME680 driver path. The spec's chip-driver structure should write read addresses with bit 7 set so reads work transparently if/when SPI is added.

### Calibration coefficient layout

Three blocks, all 8-bit registers; combine adjacent registers as the table specifies (LSB / MSB pairs are little-endian):

| Parameter | Register(s) | Type | Notes |
|-----------|-------------|------|-------|
| `par_T1`  | `0xE9` / `0xEA` | uint16 | LSB / MSB — **not** at `0x88` like BME280 |
| `par_T2`  | `0x8A` / `0x8B` | int16 | LSB / MSB |
| `par_T3`  | `0x8C`          | int8  | single byte |
| `par_P1`  | `0x8E` / `0x8F` | uint16 | |
| `par_P2`  | `0x90` / `0x91` | int16 | |
| `par_P3`  | `0x92`          | int8 | |
| `par_P4`  | `0x94` / `0x95` | int16 | |
| `par_P5`  | `0x96` / `0x97` | int16 | |
| `par_P6`  | `0x99`          | int8 | |
| `par_P7`  | `0x98`          | int8 | (note: P7 is at the **lower** address than P6) |
| `par_P8`  | `0x9C` / `0x9D` | int16 | |
| `par_P9`  | `0x9E` / `0x9F` | int16 | |
| `par_P10` | `0xA0`          | uint8 | |
| `par_H1`  | `0xE2<3:0>`, `0xE3` | uint16 | `(b[0xE3] << 4) | (b[0xE2] & 0x0F)` — 12 bits unsigned, **shares `0xE2` with `par_H2`** |
| `par_H2`  | `0xE2<7:4>`, `0xE1` | uint16 | `(b[0xE1] << 4) | (b[0xE2] >> 4)` — 12 bits unsigned |
| `par_H3`  | `0xE4` | int8 | |
| `par_H4`  | `0xE5` | int8 | |
| `par_H5`  | `0xE6` | int8 | |
| `par_H6`  | `0xE7` | uint8 | |
| `par_H7`  | `0xE8` | int8 | |
| `par_G1`  | `0xED` | int8 | |
| `par_G2`  | `0xEB` / `0xEC` | int16 | |
| `par_G3`  | `0xEE` | int8 | |
| `res_heat_val`        | `0x00` | int8 | Heater resistance correction (calibration) |
| `res_heat_range`      | `0x02<5:4>` | uint8 | Heater range — mask and shift; values 0–3 |
| `range_switching_error` | `0x04<7:4>` | int8 | Gas-range switching error — sign-extend from 4-bit signed |

Burst-read both calibration blocks (23 bytes from `0x8A` and 14 bytes from `0xE1`) plus three single-byte reads at `0x00`, `0x02`, `0x04`. The three single bytes are not in either contiguous block.

`par_H1` and `par_H2` share register `0xE2` — `par_H1` uses the lower nibble, `par_H2` uses the upper nibble. Both are 12-bit **unsigned** values stored in 16-bit slots. (This packing differs from the BME280's `dig_H4` / `dig_H5`, which are signed and use different shared bits.)

### `ctrl_hum` (`0xF2`-equivalent at `0x72`)

| Bits | Name | Description |
|------|------|-------------|
| 7    | —    | Reserved |
| 6    | `spi_3w_int_en` | SPI 3-wire interrupt enable (only meaningful when `spi3w_en = 1`) |
| 5:3  | —    | Reserved |
| 2:0  | `osrs_h` | Humidity oversampling: `000`=skipped, `001`=×1, `010`=×2, `011`=×4, `100`=×8, `101`+=×16 |

**Critical:** changes to `ctrl_hum` only take effect after a write to `ctrl_meas`. Same gotcha as BME280. Always write `ctrl_hum` first, then `ctrl_meas`.

### `ctrl_meas` (`0x74`)

| Bits | Name | Description |
|------|------|-------------|
| 7:5  | `osrs_t` | Temperature oversampling (same encoding as `osrs_h`) |
| 4:2  | `osrs_p` | Pressure oversampling (same encoding) |
| 1:0  | `mode`   | `00`=Sleep, `01`=Forced |

Only two power modes — **no normal mode**.

### `config` (`0x75`)

| Bits | Name | Description |
|------|------|-------------|
| 7:5  | —    | Reserved |
| 4:2  | `filter` | IIR filter coefficient: `000`=0, `001`=1, `010`=3, `011`=7, `100`=15, `101`=31, `110`=63, `111`=127 |
| 1    | —    | Reserved |
| 0    | `spi_3w_en` | `1` enables 3-wire SPI |

The IIR filter applies to temperature and pressure only (not humidity, not gas). Filter coefficients are different from BME280's encoding — BME680 has 8 settings up to 127, BME280 had 5 up to 16.

### `ctrl_gas_0` (`0x70`)

| Bit | Name | Description |
|-----|------|-------------|
| 3   | `heat_off` | `1` turns off the heater (override) |

### `ctrl_gas_1` (`0x71`)

| Bits | Name | Description |
|------|------|-------------|
| 4    | `run_gas`  | `1` enables gas conversion in the next forced cycle |
| 3:0  | `nb_conv`  | Index (0–9) of the heater profile to use |

### `gas_wait_x` (`0x64`–`0x6D`)

Per-profile heater on-time. Encoded as a 6-bit timer × a 2-bit multiplier:

| Bits | Name | Description |
|------|------|-------------|
| 7:6  | multiplier | `00`=×1, `01`=×4, `10`=×16, `11`=×64 |
| 5:0  | timer (1 ms units) | Value 0–63 |

Range: 1–4032 ms. The driver picks the smallest multiplier that fits the requested duration.

### `res_heat_x` (`0x5A`–`0x63`)

Per-profile target heater resistance. Computed from the desired heater temperature using `par_g1` / `par_g2` / `par_g3`, `res_heat_range`, `res_heat_val`, and an ambient temperature (the host supplies this; either a previous on-chip temperature reading or a sensible fixed value like 25 °C). Formula in the Data Conversion section.

### `meas_status_0` (`0x1D`)

| Bit | Name | Description |
|-----|------|-------------|
| 7   | `new_data` | New TPH data available since last read |
| 6   | `gas_measuring` | Gas conversion in progress |
| 5   | `measuring` | Any conversion in progress |
| 3:0 | `gas_meas_index` | Index of the most recent gas measurement (which profile fired) |

### `gas_r_lsb` (`0x2B`)

| Bits | Name | Description |
|------|------|-------------|
| 7:6  | `gas_r<1:0>` | Low 2 bits of the 10-bit gas ADC (high 8 bits in `gas_r_msb`) |
| 5    | `gas_valid_r` | `1` = real gas measurement (not a dummy slot) |
| 4    | `heat_stab_r` | `1` = heater reached its target temperature within tolerance |
| 3:0  | `gas_range_r` | Gas-range code (0–15) — selects the constant-array index for resistance conversion |

## Initialization Sequence

### Minimal

1. (Optional) Read `id` (`0xD0`) and confirm `0x61`.
2. Read calibration:
   - 23 bytes from `0x8A` (block 1).
   - 14 bytes from `0xE1` (block 2).
   - Single bytes at `0x00`, `0x02`, `0x04`.
   Unpack into the 28 calibration values in the table above.
3. Write `ctrl_hum` with `osrs_h = ×1`.
4. Write `ctrl_meas` with `osrs_t = ×1`, `osrs_p = ×1`, `mode = 00` (sleep) — must follow `ctrl_hum`.
5. Write `config` with `filter = 0` (off), `spi_3w_en = 0`.
6. Configure heater profile 0 for the default gas measurement:
   - Compute `res_heat_0` for target heater temperature 320 °C using ambient = 25 °C (initial estimate).
   - Write `gas_wait_0` for 150 ms (`gas_wait = (150 / 4) | (0x40)` → multiplier ×4, timer 37 ≈ 148 ms).
   - Set `ctrl_gas_1` = `run_gas << 4 | nb_conv` = `0x10` (run_gas=1, profile 0).

After the first temperature reading, the driver should re-compute `res_heat_0` using the actual ambient temperature and re-write that register so the heater target is more accurate.

### Full

Same as Minimal, but the user may pass `osrs_t`, `osrs_p`, `osrs_h`, `filter`, `mode`, and an arbitrary heater profile (temperature + duration + index) at construction or change them later via setters. The user can also configure all 10 heater profiles for cycling measurements.

## Implementation Stages

### Minimal

Goal: read calibrated temperature (°C), pressure (hPa), humidity (%RH), and gas resistance (Ω) with no configuration beyond the transport.

Constructor parameters: `transport`. The constructor reads and stores all 28 calibration coefficients and configures profile 0 with the default heater (320 °C / 150 ms, ambient 25 °C).

**Baked-in defaults:**
- Power mode: forced (only mode the chip supports besides sleep)
- `osrs_t = ×1`, `osrs_p = ×1`, `osrs_h = ×1`
- IIR filter off
- Gas heater profile 0: 320 °C target, 150 ms duration, gas conversion enabled (`run_gas = 1`)

| Operation             | Parameters | Returns | Notes |
|-----------------------|------------|---------|-------|
| `temperature()`       | —          | float   | °C; caches `t_fine` and updates the heater target ambient |
| `pressure()`          | —          | float   | hPa |
| `humidity()`          | —          | float   | %RH |
| `gas_resistance()`    | —          | float   | Ω; returns the most-recent reading or a fresh one if `gas_valid_r` was clear |

A single forced-mode trigger samples T, P, H, and G in sequence (TPHG cycle). The driver triggers once per call, sleeps for the worst-case measurement time (TPH conversion + heater duration + gas measurement, typically ~210 ms for the default profile), then reads all 10 output bytes (`0x1F`–`0x2B`) in one burst. Each of the four read methods triggers a fresh TPHG cycle so they are self-contained.

If `gas_valid_r` or `heat_stab_r` is `0` after a measurement, `gas_resistance()` returns `NaN` (or the language equivalent) and sets a warning flag — the heater did not stabilize in the allotted time, which usually means the duration is too short for the requested temperature.

### Full

Extends Minimal. Adds power-mode control, oversampling for all three TPH channels, IIR filter, multi-profile heater, and ambient-temperature override.

## Data Conversion

All compensation formulas are from datasheet section 3.3 — use the **integer** path (the floating-point paths are reference only). Pressure compensation is 32-bit only; humidity is 32-bit; gas resistance needs 64-bit. Temperature is 32-bit.

```
# After burst-reading registers 0x1F..0x2B (13 bytes):
press_adc = (raw[0] << 12) | (raw[1] << 4) | (raw[2] >> 4)        # 0x1F..0x21
temp_adc  = (raw[3] << 12) | (raw[4] << 4) | (raw[5] >> 4)        # 0x22..0x24
hum_adc   = (raw[6] << 8)  |  raw[7]                               # 0x25..0x26
gas_adc   = (raw[11] << 2) | (raw[12] >> 6)                        # 0x2A..0x2B<7:6>
gas_range = raw[12] & 0x0F
gas_valid = (raw[12] >> 5) & 1
heat_stab = (raw[12] >> 4) & 1

# Temperature (returns °C; updates t_fine, signed 32-bit):
var1 = (temp_adc >> 3) - (par_T1 << 1)
var2 = (var1 * par_T2) >> 11
var3 = (((var1 >> 1) * (var1 >> 1)) >> 12) * (par_T3 << 4) >> 14
t_fine = var2 + var3
T = ((t_fine * 5 + 128) >> 8) / 100.0                              # °C

# Pressure (32-bit; returns Pa):
var1 = (t_fine >> 1) - 64000
var2 = ((((var1 >> 2) * (var1 >> 2)) >> 11) * par_P6) >> 2
var2 = var2 + ((var1 * par_P5) << 1)
var2 = (var2 >> 2) + (par_P4 << 16)
var1 = (((((var1 >> 2) * (var1 >> 2)) >> 13) * (par_P3 << 5)) >> 3) + ((par_P2 * var1) >> 1)
var1 = var1 >> 18
var1 = ((32768 + var1) * par_P1) >> 15
press_comp = 1048576 - press_adc
press_comp = ((press_comp - (var2 >> 12)) * 3125)
if press_comp >= (1 << 30):
    press_comp = (press_comp // var1) << 1
else:
    press_comp = (press_comp << 1) // var1
var1 = (par_P9 * (((press_comp >> 3) * (press_comp >> 3)) >> 13)) >> 12
var2 = ((press_comp >> 2) * par_P8) >> 13
var3 = ((press_comp >> 8) * (press_comp >> 8) * (press_comp >> 8) * par_P10) >> 17
press_comp = press_comp + ((var1 + var2 + var3 + (par_P7 << 7)) >> 4)   # Pa
return press_comp / 100.0                                                # hPa

# Humidity (32-bit; returns %RH):
temp_scaled = t_fine
var1 = hum_adc - ((par_H1 << 4) + (((temp_scaled * par_H3) // 100) >> 1))
var2 = (par_H2 * (((temp_scaled * par_H4) // 100) +
                 (((temp_scaled * ((temp_scaled * par_H5) // 100)) >> 6) // 100) +
                 (1 << 14))) >> 10
var3 = var1 * var2
var4 = ((par_H6 << 7) + ((temp_scaled * par_H7) // 100)) >> 4
var5 = ((var3 >> 14) * (var3 >> 14)) >> 10
var6 = (var4 * var5) >> 1
hum_comp = (((var3 + var6) >> 10) * 1000) >> 12
return clamp(hum_comp, 0, 100000) / 1000.0                                # %RH

# Heater target resistance (run once at heater configuration):
# target_temp_c is desired heater temp; amb_temp_c is ambient (e.g. 25 or last on-chip temp)
var1 = (((amb_temp_c * par_G3) // 10) << 8)
var2 = (par_G1 + 784) * (((((par_G2 + 154009) * target_temp_c * 5) // 100) + 3276800) // 10)
var3 = var1 + (var2 >> 1)
var4 = var3 // (res_heat_range + 4)
var5 = (131 * res_heat_val) + 65536
res_heat_x100 = ((var4 // var5) - 250) * 34
res_heat_x = (res_heat_x100 + 50) // 100                                  # u8 to write to res_heat_x

# Gas-wait encoding (target_ms ∈ [1, 4032]):
if   target_ms <= 0x3F:    gw = target_ms                  # ×1, no shift
elif target_ms <= 0x3F * 4: gw = (1 << 6) | (target_ms // 4)
elif target_ms <= 0x3F * 16: gw = (2 << 6) | (target_ms // 16)
else:                       gw = (3 << 6) | min(target_ms // 64, 0x3F)

# Gas resistance (64-bit; needs the two 16-entry lookup tables from Table 16):
const_array1_int = [2147483647, 2147483647, 2147483647, 2147483647, 2147483647,
                    2126008810, 2147483647, 2130303777, 2147483647, 2147483647,
                    2143188679, 2136746228, 2147483647, 2126008810, 2147483647,
                    2147483647]
const_array2_int = [4096000000, 2048000000, 1024000000, 512000000, 255744255,
                    127110228, 64000000,    32258064,   16016016,  8000000,
                    4000000,    2000000,    1000000,    500000,    250000,
                    125000]

var1 = ((1340 + 5 * range_switching_error) * const_array1_int[gas_range]) >> 16
var2 = ((gas_adc << 15) - (1 << 24)) + var1
gas_res_ohms = ((const_array2_int[gas_range] * var1) >> 9) + (var2 >> 1)
gas_res_ohms = gas_res_ohms // var2                                       # Ω
```

The two `const_array*_int` tables must be embedded in the driver. Lengths are exactly 16; values are from datasheet Table 16 (page 23). The floating-point columns of that table are not used.

### Validation

The datasheet does not provide a worked example with explicit calibration values like the BMP280 / BME280 do. Cross-check the compensation formulas against the Bosch reference C source at <https://github.com/BoschSensortec/BME68x-Sensor-API> at known calibration values, or compare against an existing open-source library (e.g. Adafruit_BME680, pimoroni's bme680-python) at the same `*_adc` values.

A sanity check that does not require known calibration values: at room conditions (23 °C, 1013 hPa, 50 %RH), all three compensated values should be in plausible ranges; if one is wildly off (e.g. 5 °C or 200 hPa), the calibration parsing — especially the `par_H1` / `par_H2` nibble packing or the `par_T1` address — is the likely culprit.

### Altitude / sea-level / dew point

Altitude and sea-level conversions are the same international barometric formulas used by the BMP180/BMP280/BME280. Dew point uses the Magnus-Tetens approximation. See `specs/environmental/bme280.md` for the formulas; reuse them verbatim in this driver's helpers.

### Demo

A small "what is in this room?" probe. The demo polls all four sensors once every 5 seconds for 5 minutes (60 ticks). Each tick prints `T / RH / P / R_gas / dew_point / altitude` on one line. Halfway through (tick 30), the demo prompts the user to spray a small amount of isopropyl alcohol or rubbing alcohol near the sensor (or to crack a marker pen and waft it past); the user sees gas resistance drop sharply (often by 50 % or more) within 1–2 ticks and slowly recover over the remaining 30 ticks. ## Timing Constraints

- Power-on / soft-reset start-up time before first communication: 2 ms
- Maximum TPH measurement time depends on the sum of oversampling settings (very similar to BME280):

  | Preset                        | osrs_t | osrs_p | osrs_h | TPH max  |
  |-------------------------------|--------|--------|--------|----------|
  | Lowest (×1/×1/×1)             | ×1     | ×1     | ×1     | ~10 ms   |
  | Maximum (×16/×16/×16)         | ×16    | ×16    | ×16    | ~144 ms  |

- Gas measurement adds the heater-on time (`gas_wait_x`) plus a few ms of measurement. Default 150 ms heater + ~15 ms measurement → ~165 ms.
- Total forced-cycle wall clock = TPH max + gas wait + ~15 ms. For default settings this is ~190 ms; the driver should sleep for `200 ms` after triggering forced mode.
- The heater needs ~20–30 ms of the configured `gas_wait_x` to reach a 320 °C target from cold. Heating durations below 30 ms produce `heat_stab_r = 0` regardless of target temperature.
- Gas-sensor "burn-in": brand-new BME680 chips need ~5 days of continuous operation in ambient air before their gas-resistance baseline stabilizes (BSEC handles this automatically; raw readings during burn-in are not directly comparable across days).

## Implementation Notes

- **Heritage:** the BME680 driver is **not** a subclass or close sibling of the BME280. Despite the family name, the calibration parameter names, addresses, packing, compensation formulas, available power modes, and feature set all differ. Implement as an independent driver. Some helpers (altitude / dew point / IIR filter encoding) can be lifted from the BME280 driver, but the read path and compensation are wholly separate.
- **`ctrl_hum` write order** is the same gotcha as BME280: always write `ctrl_hum` (`0x72`) before `ctrl_meas` (`0x74`). Humidity oversampling otherwise never latches.
- **Calibration parsing is the dominant source of bugs.** The split into three regions (`0x8A`–`0xA0`, `0xE1`–`0xEE`, plus single bytes at `0x00`/`0x02`/`0x04`), the unusual order (`par_P7` at `0x98` is at a *lower* address than `par_P6` at `0x99`), the nibble-packed `par_H1` / `par_H2` sharing register `0xE2`, the bit-field encoding of `res_heat_range` and `range_switching_error`, and the `par_T1` location at `0xE9` (not `0x88`!) make this the most error-prone init step. Cross-check against the Bosch reference C parser before debugging anything else if humidity or gas readings look wrong.
- **Heater target requires ambient temperature.** The formula uses an ambient temperature to compute `res_heat_x`. The first time the driver computes this (in the constructor), it has no on-chip reading yet — use 25 °C as a sensible default. After the first `temperature()` call, store the latest reading and use it on subsequent `set_heater()` calls. The Full driver exposes `set_ambient_temperature()` for users who want to override.
- **Gas resistance lookup tables** must be embedded in the driver as two 16-entry constant arrays (`const_array1_int`, `const_array2_int` from datasheet Table 16, integer columns). Length is exactly 16; do not use the floating-point columns.
- **`heat_stab_r` and `gas_valid_r` matter.** A successful `gas_resistance()` reading requires both bits to be `1` in `gas_r_lsb`. If either is `0`, the heater did not stabilize within `gas_wait_x` (typically because the duration is too short for the target temperature) or the slot was a dummy. The driver should expose this via `heater_stable()` / `gas_valid()` in Full and return NaN / a sentinel from `gas_resistance()` in Minimal when either is clear.
- **Burst-read 13 output bytes** from `0x1F` to `0x2B` in one transaction. Splitting the read into multiple transactions risks tearing if a new TPHG cycle starts mid-read.
- **Single forced-mode trigger samples all four channels** when `osrs_*` and `run_gas` are non-zero. The driver triggers once per call (regardless of which value the caller requested), waits the worst-case time, then reads. Calling all four read methods in sequence costs four separate cycles; use `read_all()` from Full for tight loops.
- **Only sleep and forced modes.** Unlike BMP280 / BME280, there is no normal mode. Every reading is single-shot. Drivers that share a state-machine pattern with BME280 must drop the "are we in normal mode?" branch.
- **No 64-bit math for pressure.** Unlike BMP280 / BME280, the BME680 pressure compensation fits in 32-bit integer math by design. The gas-resistance compensation does need 64-bit; nothing else does.
- **Soft reset** (`0xB6` → `0xE0`) returns to power-on state. Calibration NVM survives but the driver must re-read it (defensive — a bad NVM read here is the kind of bug that produces silently wrong readings forever) and re-apply user configuration including the heater profile.
- **No IAQ index in this driver.** The chip outputs raw gas resistance in Ohms. Translating that to Bosch's 0–500 IAQ scale requires the closed-source BSEC software, which is out of scope. Users who want IAQ should run BSEC on top of this driver's raw readings.
- **SPI scope:** initial implementation is I²C-only. The chip driver should write read addresses with bit 7 set (the natural address) so a future register-aware SPI transport works on reads. SPI also needs page-aware logic in the BME680 driver itself (not just the transport) — defer until both pieces are in place.
- **`0x77` collides** with BMP180 / BMP280 / BME280 / BMP388. Chip-ID `0x61` distinguishes the BME680 at runtime.
