# Chip Spec: BMP280

**Manufacturer:** Bosch Sensortec  
**Datasheet:** `datasheets/pressure/bmp280.pdf`  
**Category:** `pressure`  
**Transports:** I²C, SPI (3-wire and 4-wire)

## Overview

The BMP280 is a piezo-resistive digital barometric pressure sensor with an integrated temperature sensor and both I²C and SPI interfaces. It is the function-compatible successor of the BMP180, with substantially improved performance: ±1 hPa absolute accuracy, 0.16 Pa pressure resolution and 0.01 °C temperature resolution in ultra-high-resolution mode, configurable oversampling (×1 to ×16) on each channel, a built-in IIR filter to suppress short-term disturbances, three power modes (sleep, forced, normal), and a 63 % smaller footprint than the BMP180. Pressure range is 300–1100 hPa (≈ +9000 m to −500 m), supply voltage is 1.71–3.6 V, and current consumption is 2.7 µA at 1 Hz. Bosch ships factory-calibrated trimming parameters in on-chip NVM that the host reads once and uses in a fixed compensation algorithm.

The BMP280's compensation algorithm differs from the BMP180's: 18 trimming coefficients (`dig_T1`–`dig_T3`, `dig_P1`–`dig_P9`) and 64-bit integer math (the datasheet provides a 32-bit fallback). The drivers are independent of the BMP180 driver — they share neither code nor calibration format.

## Transport Configuration

### I²C

- **Address:** `0x76` (SDO = GND) or `0x77` (SDO = V_DDIO)
- **Max clock:** 3.4 MHz (high-speed mode)
- **Pull-ups:** required on SDA and SCL; CSB must be tied to V_DDIO to select I²C mode

The 7-bit address shares its low bit with the SDO pin level. SDO must not be left floating — tie it to GND or V_DDIO. Note that `0x77` collides with the BMP180; use `0x76` if both chips share a bus.

### SPI

- **Mode:** CPOL = CPHA = 0 (mode 00) or CPOL = CPHA = 1 (mode 11). The chip auto-detects which by sampling SCK at the falling edge of CSB.
- **Max clock:** 10 MHz
- **Bit order:** MSB first
- **CS active:** low (CSB pin)
- **Wire mode:** 4-wire by default; set `config.spi3w_en = 1` for 3-wire mode (SDI is bidirectional, SDO is hi-Z)
- **Register-address protocol:** SPI uses 7-bit register addresses; bit 7 of the first byte is the R/W flag. **Read:** send `reg_addr | 0x80` (RW = 1), then read N bytes. **Write:** send `reg_addr & 0x7F` (RW = 0), then send data.

Note that **all BMP280 I²C register addresses have bit 7 set** (`0x88`–`0xFC`), which means the natural I²C register address `reg | 0x80` is what SPI uses for reads — i.e., a chip driver that reads via `transport.write_read([reg], n)` works transparently over I²C and SPI as long as the SPI transport doesn't strip bit 7. **Writes** differ between the two: I²C uses the full address, SPI uses `reg & 0x7F`. A driver that supports both must either branch on the bus type or use a register-aware SPI transport that clears bit 7 on writes automatically.

**SPI is now supported across all platforms** — `SPITransport` variants exist for MicroPython, CircuitPython, Linux (spidev), Arduino, Zephyr, and Rust (embedded-hal `SpiDevice`). See `specs/transport_spi.md`. Chip drivers may implement SPI from the initial pass; see the bit-7 write-address caveat in Implementation Notes.

| Address    | Name         | R/W | Reset    | Description |
|------------|--------------|-----|----------|-------------|

| `0xD0`     | `id`         | R   | `0x58`  | Fixed chip-ID; presence check |

| `0xF3`     | `status`     | R   | `0x00`  | Bit 3 = `measuring`, bit 0 = `im_update` |
| `0xF4`     | `ctrl_meas`  | R/W | `0x00`  | `osrs_t<7:5>`, `osrs_p<4:2>`, `mode<1:0>` |
| `0xF5`     | `config`     | R/W | `0x00`  | `t_sb<7:5>`, `filter<4:2>`, `spi3w_en<0>` |
| `0xF7`     | `press_msb`  | R   | `0x80`  | Pressure ADC bits 19:12 |
| `0xF8`     | `press_lsb`  | R   | `0x00`  | Pressure ADC bits 11:4 |
| `0xF9`     | `press_xlsb` | R   | `0x00`  | Pressure ADC bits 3:0 in `[7:4]` |
| `0xFA`     | `temp_msb`   | R   | `0x80`  | Temperature ADC bits 19:12 |
| `0xFB`     | `temp_lsb`   | R   | `0x00`  | Temperature ADC bits 11:4 |

A reset value of `0x80` for the MSB output registers means "no measurement yet" — readings are valid only after the first measurement completes.

### Calibration NVM (`0x88`–`0x9F`)

| LSB / MSB         | Name    | Type   |
|-------------------|---------|--------|
| `0x88` / `0x89`   | dig_T1  | uint16 |
| `0x8A` / `0x8B`   | dig_T2  | int16  |
| `0x8C` / `0x8D`   | dig_T3  | int16  |
| `0x8E` / `0x8F`   | dig_P1  | uint16 |
| `0x90` / `0x91`   | dig_P2  | int16  |
| `0x92` / `0x93`   | dig_P3  | int16  |
| `0x94` / `0x95`   | dig_P4  | int16  |
| `0x96` / `0x97`   | dig_P5  | int16  |
| `0x98` / `0x99`   | dig_P6  | int16  |
| `0x9A` / `0x9B`   | dig_P7  | int16  |
| `0x9C` / `0x9D`   | dig_P8  | int16  |
| `0x9E` / `0x9F`   | dig_P9  | int16  |

**Calibration NVM is little-endian** (LSB at the lower register address) — opposite of the BMP180 and of the data registers. Use `struct.unpack('<HhhHhhhhhhhh', raw)` (Python) / `Buffer.readUInt16LE` / `Buffer.readInt16LE` (Node.js) / `i16::from_le_bytes` (Rust) when parsing the 24-byte block.

### `ctrl_meas` (`0xF4`)

| Bits | Name    | Description |
|------|---------|-------------|
| 7:5  | `osrs_t` | Temperature oversampling: `000`=skipped, `001`=×1, `010`=×2, `011`=×4, `100`=×8, `101`–`111`=×16 |
| 4:2  | `osrs_p` | Pressure oversampling: same encoding (`001`–`101`+) |
| 1:0  | `mode`   | `00`=Sleep, `01`=Forced, `10`=Forced, `11`=Normal |

Skipping (`000`) sets the corresponding output register to `0x80000`. Writing any forced-mode value triggers a single-shot conversion; the chip auto-returns to sleep when done.

### `config` (`0xF5`)

| Bits | Name      | Description |
|------|-----------|-------------|
| 7:5  | `t_sb`    | Standby time in normal mode (see table below) |
| 4:2  | `filter`  | IIR filter coefficient (`000`=off, `001`=2, `010`=4, `011`=8, `100`–`111`=16) |
| 1    | —         | Reserved |
| 0    | `spi3w_en` | `1` enables 3-wire SPI |

**`t_sb` (standby time in normal mode):**

| Value | t_standby |
|-------|-----------|
| 000   | 0.5 ms |
| 001   | 62.5 ms |
| 010   | 125 ms |
| 011   | 250 ms |
| 100   | 500 ms |
| 101   | 1000 ms |
| 110   | 2000 ms |
| 111   | 4000 ms |

Writes to `config` while in normal mode may be ignored — the chip recommends entering sleep first if you change the IIR filter or standby on the fly.

### `status` (`0xF3`)

| Bit | Name        | Description |
|-----|-------------|-------------|
| 3   | `measuring` | `1` while a conversion is running |
| 0   | `im_update` | `1` while NVM is being copied to image registers (briefly during reset and at the start of every conversion) |

## Initialization Sequence

### Minimal

1. (Optional) Read `id` (`0xD0`) and confirm `0x58`. Recommended.
2. Burst-read 24 bytes from `0x88`–`0x9F` and unpack into 12 trimming coefficients (little-endian).
3. Write `ctrl_meas` (`0xF4`) with `osrs_t = ×1`, `osrs_p = ×1`, `mode = 00` (sleep) — leaves the chip in sleep until the first measurement is requested.
4. (Recommended) Write `config` (`0xF5`) with `filter = off`, `t_sb` = don't care, `spi3w_en = 0`.

The driver triggers each measurement explicitly in **forced mode** by writing `ctrl_meas` with `mode = 01` and then polling `status` (or sleeping for the maximum measurement time).

### Full

Same as Minimal, but the user may pass `osrs_t`, `osrs_p`, `filter`, `t_sb`, and `mode` to the constructor or change them later via `configure()` / `set_*()`. The driver writes the chosen values once on init and again whenever the user calls a setter.

## Implementation Stages

### Minimal

Goal: read calibrated temperature (°C) and pressure (hPa) with no configuration beyond the transport.

Constructor parameters: `transport`. The constructor reads and stores all 12 trimming coefficients.

**Baked-in defaults:**
- Power mode: forced (each `temperature()` / `pressure()` call triggers a single-shot conversion)
- `osrs_t = ×1`, `osrs_p = ×1` (Ultra Low Power)
- IIR filter off
- `spi3w_en = 0`

| Operation       | Parameters | Returns | Notes |
|-----------------|------------|---------|-------|
| `temperature()` | —          | float   | °C; caches `t_fine` for use in subsequent pressure compensation |
| `pressure()`    | —          | float   | hPa; internally re-triggers a forced measurement and re-reads both UT and UP, then refreshes `t_fine` |

Per the datasheet measurement-cycle figure, a single forced-mode write performs *both* temperature and pressure conversions in sequence (when both `osrs_t` and `osrs_p` are non-zero), and the driver reads all 6 result bytes in one burst (`press_msb`–`temp_xlsb`, registers `0xF7`–`0xFC`). So `pressure()` and `temperature()` cost the same: ~6.4 ms wall clock at the default oversampling.

### Full

Extends Minimal. Adds power-mode control, oversampling, IIR filter, standby, and altitude helpers.

| Operation                  | Parameters                                              | Returns | Notes |
|----------------------------|---------------------------------------------------------|---------|-------|
| *(inherits Minimal)*       | | | |
| `configure()`              | `osrs_t`, `osrs_p`, `mode`, `filter`, `t_sb`            | —       | Writes both `ctrl_meas` and `config` in one call |
| `set_oversampling()`       | `osrs_t`, `osrs_p`                                      | —       | Update `osrs_t` / `osrs_p` bits of `ctrl_meas` |
| `set_mode()`               | `mode`                                                  | —       | Update `mode` bits of `ctrl_meas` |
| `set_filter()`             | `coeff`                                                 | —       | Update `filter` bits of `config` |
| `set_standby()`            | `t_sb`                                                  | —       | Update `t_sb` bits of `config` (only relevant in Normal mode) |
| `status()`                 | —                                                       | int     | Read `0xF3`; bits `MEASURING` (`0x08`) and `IM_UPDATE` (`0x01`) |
| `altitude()`               | `sea_level_hpa=1013.25`                                 | float   | Meters above sea level computed from the current pressure |
| `sea_level_pressure()`     | `altitude_m`                                            | float   | Sea-level pressure (hPa) implied by current pressure and a known altitude |
| `chip_id()`                | —                                                       | int     | Read `0xD0`; expect `0x58` |
| `reset()`                  | —                                                       | —       | Soft reset (write `0xB6` to `0xE0`); re-reads calibration and re-applies user configuration |

**Constants exposed alongside the Full class:**

Oversampling: `OSRS_SKIP = 0`, `OSRS_X1 = 1`, `OSRS_X2 = 2`, `OSRS_X4 = 3`, `OSRS_X8 = 4`, `OSRS_X16 = 5`  
Mode: `MODE_SLEEP = 0`, `MODE_FORCED = 1`, `MODE_NORMAL = 3`  
Filter: `FILTER_OFF = 0`, `FILTER_2 = 1`, `FILTER_4 = 2`, `FILTER_8 = 3`, `FILTER_16 = 4`  
Standby: `T_SB_0_5_MS = 0`, `T_SB_62_5_MS = 1`, `T_SB_125_MS = 2`, `T_SB_250_MS = 3`, `T_SB_500_MS = 4`, `T_SB_1000_MS = 5`, `T_SB_2000_MS = 6`, `T_SB_4000_MS = 7`  
Status flags: `STATUS_MEASURING = 0x08`, `STATUS_IM_UPDATE = 0x01`

When `mode = MODE_NORMAL`, the chip cycles continuously between conversions and standby. In that mode, `temperature()` and `pressure()` should **not** trigger a new forced measurement — they should read the most recent shadow registers. Drivers track the active mode and adapt the read path accordingly. (When `mode = MODE_SLEEP` or `MODE_FORCED`, calls re-trigger.)

## Data Conversion

The compensation algorithm is from datasheet section 3.11.3 and **must** be implemented exactly as documented, using 64-bit integer arithmetic. A 32-bit fallback (datasheet Appendix 8.2) is acceptable on platforms without 64-bit integers, but loses some precision.

```
# After burst-reading registers 0xF7..0xFC (6 bytes):
adc_P = (raw[0] << 12) | (raw[1] << 4) | (raw[2] >> 4)   # 20-bit unsigned
adc_T = (raw[3] << 12) | (raw[4] << 4) | (raw[5] >> 4)   # 20-bit unsigned

# Temperature (returns °C as 1/100 °C in t_fine; T returned in °C):
var1   = (((adc_T >> 3) - (dig_T1 << 1)) * dig_T2) >> 11
var2   = (((((adc_T >> 4) - dig_T1) * ((adc_T >> 4) - dig_T1)) >> 12) * dig_T3) >> 14
t_fine = var1 + var2
T      = ((t_fine * 5 + 128) >> 8) / 100.0               # °C

# Pressure (64-bit path; returns Pa in Q24.8 fixed point):
var1 = t_fine - 128000
var2 = var1 * var1 * dig_P6
var2 = var2 + ((var1 * dig_P5) << 17)
var2 = var2 + (dig_P4 << 35)
var1 = ((var1 * var1 * dig_P3) >> 8) + ((var1 * dig_P2) << 12)
var1 = (((1 << 47) + var1) * dig_P1) >> 33
if var1 == 0:
    return 0     # avoid divide-by-zero before calibration is loaded
p    = 1048576 - adc_P
p    = (((p << 31) - var2) * 3125) // var1
var1 = (dig_P9 * (p >> 13) * (p >> 13)) >> 25
var2 = (dig_P8 * p) >> 19
p    = ((p + var1 + var2) >> 8) + (dig_P7 << 4)          # Pa in Q24.8

return (p / 256.0) / 100.0                                # hPa
```

`t_fine` (signed 32-bit) is the bridge between the two compensations and must be cached on the driver between `temperature()` and `pressure()` calls. Both Minimal `pressure()` and Full `pressure()` re-read the temperature ADC alongside the pressure ADC (they are sampled in the same forced measurement cycle), so `t_fine` is always fresh.

### Validation

The datasheet (Figure on page 23) provides sample trimming values and sample ADC values:

```
dig_T1=27504, dig_T2=26435, dig_T3=-1000
dig_P1=36477, dig_P2=-10685, dig_P3=3024, dig_P4=2855, dig_P5=140,
dig_P6=-7, dig_P7=15500, dig_P8=-14600, dig_P9=6000
UT=519888, UP=415148
→ T  = 25.08 °C  (int64 result: 2508 in 1/100 °C)
→ p  = 100653 Pa = 1006.53 hPa  (int64 result: 25767236 / 256 = 100653.27 Pa)
```

Use this as a unit test for the compensation formulas before going anywhere near hardware.

### Altitude / sea-level conversions

Same as the BMP180 (international barometric formula):

```
altitude_m    = 44330 * (1 - (p_hpa / sea_level_hpa) ** (1 / 5.255))
sea_level_hpa = p_hpa / (1 - altitude_m / 44330) ** 5.255
```

A 1 hPa change at sea level corresponds to ~8.43 m altitude.

## Timing Constraints

- Power-on / soft-reset start-up time before first communication: 2 ms (much faster than BMP180's 10 ms)
- Forced-mode measurement times (typ / max), all-channel:

  | Oversampling preset | osrs_p | osrs_t | typ | max |
  |---------------------|--------|--------|-----|-----|
  | Ultra Low Power     | ×1     | ×1     | 5.5 ms  | 6.4 ms |
  | Low Power           | ×2     | ×1     | 7.5 ms  | 8.7 ms |
  | Standard Resolution | ×4     | ×1     | 11.5 ms | 13.3 ms |
  | High Resolution     | ×8     | ×1     | 19.5 ms | 22.5 ms |
  | Ultra High Resolution | ×16  | ×2     | 37.5 ms | 43.2 ms |

  Drivers should sleep for the **max** time after triggering forced mode, then read the data registers. Polling `status.measuring` is an alternative for shorter wait times but requires extra I²C traffic.
- Normal-mode standby times: see the `t_sb` table above.
- The IIR filter's settling time is `samples_to_75% × t_meas`: for `filter = 16`, that is 22 samples × t_meas. At ×16/×2 / t_sb = 0.5 ms (~26 Hz ODR), the filter takes ~0.85 s to converge after a step.

## Implementation Notes

- **Calibration NVM is little-endian**, the data registers are not. This is the most common bug when porting BMP180 code to BMP280 — the parsing format is reversed, and the coefficient names changed.
- The compensation algorithm needs **64-bit integer** intermediates. On Python and Node.js, native integers handle this. On C++ and Rust, use `int64_t` / `i64`. A 32-bit fallback exists (datasheet Appendix 8.2) and may be used on resource-constrained platforms — but the algorithm differs and produces slightly different output. Choose one path and stick with it; the 64-bit path is the recommended default.
- `t_fine` from the temperature compensation is shared with the pressure compensation. The driver caches `t_fine` after each temperature read (which happens on every `temperature()` AND `pressure()` call, since both ADCs are sampled in the same forced cycle).
- A **single forced-mode trigger samples both ADCs** when both `osrs_t` and `osrs_p` are non-zero. The driver reads all 6 output bytes in one burst from `0xF7`. There is no need for separate "trigger temperature" and "trigger pressure" steps like the BMP180 has.
- In **normal mode**, the chip continuously cycles. Reading the data registers returns the most recent shadow values. The driver should track which mode it is in: in forced/sleep mode, `temperature()` / `pressure()` triggers a new conversion + waits + reads; in normal mode, they just read.
- **Burst-read the 6 output registers in one transaction** to avoid the data-register-shadowing race condition (datasheet section 3.10). Reading registers individually can mix bytes from two different measurement cycles in normal mode.
- **SPI register-protocol caveat:** all BMP280 register addresses have bit 7 set. For SPI **reads**, the chip already wants bit 7 = 1, so `transport.write_read([reg], n)` works on both buses if the chip driver passes the natural address. For SPI **writes**, the driver must clear bit 7 (`reg & 0x7F`). A driver that supports both buses needs a `bus_type` constructor argument (or equivalent) so it knows when to apply the mask. For I²C-only drivers, the natural register address works unchanged.
- **`0x77` collides with the BMP180.** If both chips are on one bus, set the BMP280's SDO to GND for `0x76`.
- The `id` register (`0x58`) is the only programmatic way to confirm a BMP280 versus a different Bosch sensor (the BME280 reads `0x60`, the BMP388 reads `0x50`). Drivers should verify on init.
- Soft reset (`0xB6` → `0xE0`) takes the chip back to power-on state. Calibration NVM survives, but the driver must re-read it (defensive — costs little and verifies the chip came back) and re-apply any user-configured `ctrl_meas` / `config` values that differ from the power-on defaults.
- The `status.im_update` bit is set briefly after every conversion while the NVM image is refreshed. It is harmless to read data registers during that window — the data registers contain the previous reading until shadowing completes.
