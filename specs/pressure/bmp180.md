# Chip Spec: BMP180

**Manufacturer:** Bosch Sensortec  
**Datasheet:** `datasheets/pressure/bmp180.pdf`  
**Category:** `pressure`  
**Transports:** I²C

## Overview

The BMP180 is a piezo-resistive digital barometric pressure sensor with an integrated temperature sensor and an I²C interface. Pressure range is 300–1100 hPa (about +9000 m to −500 m relative to sea level), with output resolution down to 0.01 hPa and four selectable oversampling modes that trade conversion time for noise. The chip ships with 11 factory-calibrated coefficients in on-board EEPROM that the host must read once at startup and use in a fixed compensation algorithm to convert raw ADC readings into Pa and °C. Operates from 1.62–3.6 V (V_DDIO), draws 3–32 µA at 1 sample/s depending on mode, and starts up in 10 ms. The function-compatible successor of the BMP085.

## Transport Configuration

### I²C

- **Address:** `0x77` (fixed — no programmable address pin)
- **Max clock:** 3.4 MHz (high-speed mode)
- **Pull-ups:** 2.2 kΩ – 10 kΩ to V_DDIO (4.7 kΩ typical)

The datasheet quotes the address as `0xEE` (write) / `0xEF` (read); these are the 8-bit forms with the R/W bit appended. The 7-bit address used by all standard host APIs is `0x77`. Drivers should not expose the address as a constructor argument, since there is exactly one possible value.

All 16-bit registers are big-endian (MSB at the lower register address). Reads can be done as a multi-byte burst from a single starting register address — the BMP180 auto-increments the internal pointer.

## Register Map

| Address | Name      | R/W | Reset | Description |
|---------|-----------|-----|-------|-------------|
| `0xAA`–`0xBF` | Calibration EEPROM | R | factory | 11 × 16-bit calibration coefficients |
| `0xD0`  | `id`      | R   | `0x55`| Fixed chip-ID; presence check |
| `0xE0`  | `soft_reset` | W | —    | Write `0xB6` to perform a power-on reset |
| `0xF4`  | `ctrl_meas` | R/W | `0x00` | Measurement control: `oss<7:6>`, `sco<5>`, `meas<4:0>` |
| `0xF6`  | `out_msb`   | R   | `0x80` | ADC result bits 23:16 (or 15:8 for 16-bit mode) |
| `0xF7`  | `out_lsb`   | R   | `0x00` | ADC result bits 15:8 (or 7:0) |
| `0xF8`  | `out_xlsb`  | R   | `0x00` | ADC result bits 7:5 (only valid bits 7:5; only used when oss=3) |

### Calibration EEPROM (`0xAA`–`0xBF`)

| Parameter | MSB addr | LSB addr | Type   |
|-----------|----------|----------|--------|
| AC1       | `0xAA`   | `0xAB`   | int16  |
| AC2       | `0xAC`   | `0xAD`   | int16  |
| AC3       | `0xAE`   | `0xAF`   | int16  |
| AC4       | `0xB0`   | `0xB1`   | uint16 |
| AC5       | `0xB2`   | `0xB3`   | uint16 |
| AC6       | `0xB4`   | `0xB5`   | uint16 |
| B1        | `0xB6`   | `0xB7`   | int16  |
| B2        | `0xB8`   | `0xB9`   | int16  |
| MB        | `0xBA`   | `0xBB`   | int16  |
| MC        | `0xBC`   | `0xBD`   | int16  |
| MD        | `0xBE`   | `0xBF`   | int16  |

Sanity check: none of the 11 words may read as `0x0000` or `0xFFFF`. If any one does, the I²C link is broken or the chip is faulty — fail initialization rather than continuing with garbage coefficients.

### `ctrl_meas` (`0xF4`)

| Bits | Name | Description |
|------|------|-------------|
| 7:6  | `oss` | Pressure oversampling (`00`=ULP, `01`=Standard, `10`=High Res, `11`=Ultra High Res) |
| 5    | `sco` | Start-of-conversion. Set by host on write; cleared by chip when conversion is complete |
| 4:0  | `meas` | `0x0E` (bits 01110) for temperature; `0x14` (bits 10100) for pressure |

The full byte values written to `ctrl_meas` are:

| Measurement       | Value | Max conversion time |
|-------------------|-------|---------------------|
| Temperature       | `0x2E` | 4.5 ms |
| Pressure (oss=0)  | `0x34` | 4.5 ms |
| Pressure (oss=1)  | `0x74` | 7.5 ms |
| Pressure (oss=2)  | `0xB4` | 13.5 ms |
| Pressure (oss=3)  | `0xF4` | 25.5 ms |

(`0x34 | (oss << 6)` is the formula.)

### Output registers (`0xF6`, `0xF7`, `0xF8`)

For temperature reads, `out_msb`/`out_lsb` hold a 16-bit unsigned value `UT`. For pressure reads, the 19-bit value is assembled as:

```
UP = ((MSB << 16) | (LSB << 8) | XLSB) >> (8 - oss)
```

For oss=0 the XLSB byte is unused (effectively a 16-bit read).

## Initialization Sequence

### Minimal

1. (Optional) Read `id` (`0xD0`) and confirm `0x55`. Recommended as a presence check.
2. Read all 22 calibration bytes (`0xAA`–`0xBF`) in one burst; unpack into 11 named coefficients.
3. Sanity-check: no coefficient is `0x0000` or `0xFFFF`.
4. Pick `oss = 0` (Ultra Low Power) as the default oversampling mode.

There is no "continuous mode" to enable — every `temperature()` and `pressure()` call triggers a fresh single-shot conversion.

### Full

Same as Minimal, but the user may pass a different default `oss` to the constructor, or change it later via `set_oversampling()`.

## Implementation Stages

### Minimal

Goal: read calibrated temperature (°C) and pressure (hPa) with no configuration beyond the transport.

Constructor parameters: `transport`. The constructor reads and stores all 11 calibration coefficients.

**Baked-in defaults:**
- `oss = 0` (Ultra Low Power, 4.5 ms conversion)

| Operation       | Parameters | Returns | Notes |
|-----------------|------------|---------|-------|
| `temperature()` | —          | float   | °C; caches `B5` for use in subsequent pressure compensation |
| `pressure()`    | —          | float   | hPa; internally re-reads UT to refresh B5 (self-contained) |

`pressure()` always triggers a fresh UT read first (see the datasheet measurement flow), so it can be called without first calling `temperature()`. Total wall-clock cost per `pressure()` call: 4.5 ms (UT) + conversion time for the active OSS.

### Full

Extends Minimal. Adds oversampling control and altitude helpers.

| Operation                  | Parameters       | Returns | Notes |
|----------------------------|------------------|---------|-------|
| *(inherits Minimal)*       | | | |
| `oversampling()`           | —                | int     | Current OSS (0–3) |
| `set_oversampling()`       | `oss` (0–3)      | —       | Change OSS for subsequent `pressure()` calls |
| `altitude()`               | `sea_level_hpa=1013.25` | float | Meters above sea level, computed from current pressure |
| `sea_level_pressure()`     | `altitude_m`     | float   | Sea-level pressure (hPa) implied by current pressure and a known altitude |
| `chip_id()`                | —                | int     | Read `0xD0`; expect `0x55` |
| `reset()`                  | —                | —       | Soft reset (write `0xB6` to `0xE0`); re-reads calibration coefficients afterward |

**OSS constants** exposed alongside the Full class: `OSS_ULP = 0`, `OSS_STANDARD = 1`, `OSS_HIGH_RES = 2`, `OSS_ULTRA_HIGH_RES = 3`. Each language picks its idiomatic spelling.

`altitude()` and `sea_level_pressure()` both call `pressure()` internally — they are convenience wrappers, not extra reads.

## Data Conversion

The compensation algorithm is from the datasheet (Figure 4, page 15) and **must** be implemented exactly as documented, using integer arithmetic. Floating point in the intermediate steps will produce subtly wrong results.

```
# After reading UT and UP with the active OSS:

# Temperature (true T in 0.1 °C, then convert to °C):
X1 = ((UT - AC6) * AC5) >> 15
X2 = (MC << 11) // (X1 + MD)
B5 = X1 + X2                                   # cached for pressure
T  = ((B5 + 8) >> 4) / 10.0                    # °C

# Pressure (true p in Pa, then convert to hPa):
B6 = B5 - 4000
X1 = (B2 * ((B6 * B6) >> 12)) >> 11
X2 = (AC2 * B6) >> 11
X3 = X1 + X2
B3 = (((AC1 * 4 + X3) << oss) + 2) >> 2
X1 = (AC3 * B6) >> 13
X2 = (B1 * ((B6 * B6) >> 12)) >> 16
X3 = ((X1 + X2) + 2) >> 2
B4 = (AC4 * (X3 + 32768)) >> 15                # B4 is unsigned
B7 = (UP - B3) * (50000 >> oss)                # B7 is unsigned

if B7 < 0x80000000:
    p = (B7 * 2) // B4
else:
    p = (B7 // B4) * 2

X1 = (p >> 8) * (p >> 8)
X1 = (X1 * 3038) >> 16
X2 = (-7357 * p) >> 16
p  = p + ((X1 + X2 + 3791) >> 4)               # Pa

return p / 100.0                               # hPa
```

The signed/unsigned distinctions matter on languages with fixed-width integers (C++, Rust). In Python and JS (arbitrary-precision integers), use straightforward integer math but watch for the right-shift of negative numbers — explicitly use floor division (`//` in Python, `Math.floor(x / 2**n)` in JS) where the datasheet specifies arithmetic shift right, since some languages' `>>` operators differ. Validate with the datasheet's worked example: `UT=27898`, `UP=23843`, `oss=0` → `T=15.0 °C`, `p=69964 Pa` (using AC1=408, AC2=−72, AC3=−14383, AC4=32741, AC5=32757, AC6=23153, B1=6190, B2=4, MB=−32768, MC=−8711, MD=2868).

### Altitude / sea-level conversions

```
altitude_m       = 44330 * (1 - (p_hpa / sea_level_hpa) ** (1 / 5.255))
sea_level_hpa    = p_hpa / (1 - altitude_m / 44330) ** 5.255
```

These are the international barometric formulas from datasheet sections 3.6–3.7. A 1 hPa change at sea level corresponds to ~8.43 m altitude.

| Input trigger | Output `msg.payload` fields                                  | Notes |
|---------------|--------------------------------------------------------------|-------|
| any message   | `{ temperature: degC, pressure: hPa, altitude: m }`          | Altitude computed against `sea_level_hpa` from the config panel |

Config panel fields:
- I²C bus number (Linux `/dev/i2c-N`)
- Oversampling mode (dropdown: ULP / Standard / High Res / Ultra High Res; default ULP)
- Sea-level reference pressure (hPa, default `1013.25`)

(No I²C address field — the BMP180 has a fixed `0x77`.)

### Demo flow

A Node-RED inject node fires every 5 seconds into the `periph-bmp180` node. The output goes to (a) a debug node that shows the raw payload, and (b) three `ui_chart` nodes plotting temperature, pressure, and altitude over time. A second inject node ("Calibrate to here") feeds the current pressure into a function node that recomputes `sea_level_hpa` and updates the BMP180 node's config — the user clicks it once at a known altitude (e.g., their desk) to zero the altimeter.

## Examples

### Demo

A pocket altimeter / weather logger. The demo reads temperature, pressure, and altitude once per second for 60 seconds, with the ULP oversampling mode (fastest). The first reading is used as the sea-level reference, so altitude starts at exactly 0 m and any subsequent change is relative — pick the chip up to your eye level and the demo prints "you moved up X cm" using the altitude delta. After the run, print the min/max/mean of all three readings. Demonstrates that even a 1 hPa change produces ~8 m of altitude resolution and that the chip can detect roughly 25 cm changes in still air.

## Timing Constraints

- Power-up / soft-reset start-up time before first communication: 10 ms
- Conversion times (max): 4.5 / 7.5 / 13.5 / 25.5 ms for OSS = 0 / 1 / 2 / 3
- Temperature conversion is always 4.5 ms regardless of OSS
- Sampling rate ceiling in standard mode: 128 samples/s (current scales linearly with rate)
- The datasheet recommends measuring temperature only once per second when sampling pressure faster than that, but this driver always pairs them — if you need higher throughput, drive the chip directly via the transport

## Implementation Notes

- The BMP180 has **no continuous mode**. Every reading is a single-shot triggered conversion: write `ctrl_meas` → wait the conversion time → read `out_msb`/`out_lsb`/`out_xlsb`. The driver hides this; users just call `temperature()` or `pressure()`.
- Reading all 22 calibration bytes in one burst (`write_read([0xAA], 22)`) is significantly faster than 11 separate 2-byte reads. Drivers should use the burst form.
- The compensation algorithm uses fixed-width integer arithmetic. On C++ and Rust, use `int32_t` / `i32` for the intermediate values; the datasheet's "long" type is 32-bit. `B4` and `B7` are explicitly unsigned 32-bit. On Python and Node.js, regular integer math is fine but right-shift of negative numbers must use arithmetic shift semantics (Python's `>>` is correct; JS's `>>` is also arithmetic, but `>>>` is logical — use `>>`).
- The pressure compensation needs `B5` from the most recent temperature reading. Both Minimal `pressure()` and Full `pressure()` re-read UT every call, refreshing `B5` automatically. Calling `temperature()` also refreshes `B5` as a side effect.
- `out_xlsb` (`0xF8`) is only meaningful when OSS=3. For OSS<3, drivers can either burst-read 3 bytes and let the formula `>> (8-oss)` handle masking, or burst-read 2 bytes when OSS=0/1/2. Both work.
- The `id` register is the only way to verify the chip is alive before reading calibration data. Recommend reading it in the constructor and failing fast on mismatch.
- A "fully calibrated" datasheet means each chip has unique coefficients programmed at the factory; there is no user calibration step. Coefficients of `0x0000` or `0xFFFF` indicate a damaged / disconnected chip, not a valid factory value — fail init in that case.
- The fixed `0x77` address means **only one BMP180 can live on a single I²C bus**. If two are needed, they must be on separate buses or behind a TCA9548-style multiplexer. The driver does not handle this; it is documented in the datasheet and worth flagging in the example comment if the demo could otherwise mislead.
- Soft reset (`0xB6` → `0xE0`) takes the chip back to power-on state; calibration coefficients in EEPROM survive but the cached `B5` in the driver does not. Drivers should re-read calibration after `reset()` (defensive — coefficients should not change, but a fresh read costs little and verifies the chip came back).
