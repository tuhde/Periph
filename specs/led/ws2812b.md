# Chip Spec: WS2812B

**Manufacturer:** Worldsemi  
**Datasheet:** `datasheets/led/WS2812B.pdf`  
**Category:** led  
**Transports:** NeoPixel (WS2812B single-wire NZR via SPI bit-encoding)

## Overview

The WS2812B is an addressable RGB LED with an integrated control IC packaged in a 5050 SMD component. Each pixel contains a red, green, and blue LED plus a driver IC that latches its own 24-bit GRB value from the single-wire data stream and forwards the remaining bits to the next pixel in the chain. Pixels cascade indefinitely via DIN → DOUT connections; the host only needs one GPIO or SPI-MOSI line. Each channel has 256 brightness levels, yielding 16.7 million colors per pixel at a refresh rate of at least 400 Hz. A reset pulse (≥50 µs low) latches the transmitted data into all pixels simultaneously.

The driver in this library sits on top of the NeoPixel transport (`specs/transport_neopixel.md`), which handles all timing-critical SPI bit-encoding. The chip driver is responsible only for maintaining the pixel buffer, converting RGB → GRB order, and calling `transport.write()`.

## Transport Configuration

### NeoPixel

- **Encoding:** SPI bit-encoding at 2.4 MHz (see `specs/transport_neopixel.md`)
- **Bit order:** MSB-first per byte; channel order is GRB (Green, Red, Blue)
- **Frame format:** `n × 3` bytes (24 bits per pixel), followed by ≥50 µs reset (16 SPI zero-bytes encoded by transport)
- **Max pixels at 30 fps:** 1024 (limited by data rate, not by any register)

## Pixel Data Format

No register map — the WS2812B has no addressable registers. Data is a continuous bit stream.

### 24-bit pixel word (MSB-first)

| Bits 23–16 | Bits 15–8 | Bits 7–0 |
|------------|-----------|----------|
| Green (G7–G0) | Red (R7–R6) | Blue (B7–B0) |

The driver accepts RGB from the user and reorders to GRB internally before passing to the transport.

## Initialization Sequence

1. Construct the NeoPixel transport for the target platform (see `specs/transport_neopixel.md`).
2. Construct `WS2812BMinimal(transport, n)` where `n` is the number of pixels. The constructor allocates an internal buffer of `n × 3` zero-bytes (all pixels off).
3. No further initialization is required — the strip powers up in an undefined state and can be set at any time.

## Implementation Stages

### Minimal

Goal: light up an entire strip with one color in three lines — construct, fill, done. No brightness control or per-pixel addressing required.

| Operation | Parameters | Returns | Notes |
|-----------|------------|---------|-------|
| `init` | `transport`, `n: int` | — | Allocates `n × 3` zero-byte buffer; stores transport |
| `fill` | `r: int`, `g: int`, `b: int` | — | Fills every pixel with (r, g, b); clamps each to [0, 255]; sends immediately |
| `off` | — | — | Equivalent to `fill(0, 0, 0)`; turns the entire strip off |

**Sensible defaults:** Buffer initialized to all zeros (all pixels off). No brightness scaling. `fill()` always calls the transport write so there is no separate show step.

### Full

Goal: expose per-pixel addressing, explicit frame control, global brightness scaling, and convenience helpers. Extends Minimal.

| Operation | Parameters | Returns | Notes |
|-----------|------------|---------|-------|
| *(inherits Minimal)* | | | |
| `set_pixel` | `index: int`, `r: int`, `g: int`, `b: int` | — | Write one pixel into the buffer (0-indexed); clamps to [0, 255]; does **not** send |
| `set_pixels` | `colors: list[tuple[int,int,int]]` | — | Write a list of `(r, g, b)` tuples into the buffer starting at pixel 0; does not send |
| `show` | — | — | Send the current buffer (with brightness applied) to the strip |
| `brightness` | `value: int` (0–255, get/set) | `int` | Global brightness scalar; default 255 (full); applied at `show()` time: `sent = stored × brightness // 255` |
| `rotate` | `steps: int` (default 1) | — | Shift the pixel buffer left by `steps` positions; wraps around; does not send |
| `fill_hsv` | `h: float` (0.0–1.0), `s: float` (0.0–1.0), `v: float` (0.0–1.0) | — | Convert HSV to RGB, fill every pixel, then send immediately |

**Additional configuration options:**
- Per-pixel addressing via `set_pixel()` and `set_pixels()`
- Explicit frame control: build frames with `set_pixel()` / `set_pixels()`, then push with `show()`
- Global brightness scaling (0–255) applied non-destructively at `show()` time
- Pixel buffer rotation for chase/scroll effects
- HSV fill convenience for hue-sweep animations

## Data Conversion

```
# RGB → GRB reorder (done internally before transport.write())
grb_buffer[i*3 + 0] = g    # green
grb_buffer[i*3 + 1] = r    # red
grb_buffer[i*3 + 2] = b    # blue

# Brightness scaling (applied at show() time in Full)
sent = stored_channel_value * brightness // 255

# HSV → RGB (standard algorithm, all inputs 0.0–1.0, outputs 0–255 int)
# Use platform math library or implement inline for embedded targets.
```

### Demo flow

An Inject node fires every 2 seconds with a different payload cycling through six colors (red, orange, yellow, green, blue, violet). Each payload is `{ r, g, b }`. The `periph-ws2812b` node updates the strip. A second Inject node sends `{ command: "off" }` on demand. The flow demonstrates basic color cycling on a physical strip.

## Examples

### Demo

Animate a color-wheel sweep across the strip: each frame shifts the hue offset by one step so the rainbow rotates around the strip. Use 30 pixels (configurable), update at approximately 30 fps (33 ms sleep), and print the current hue offset each second. After 10 seconds of rainbow rotation, trigger a strobe effect (alternate full white / off at 10 Hz for 2 seconds), then return to the rainbow. This exercises the brightness property, per-pixel addressing, and timing for a visually engaging real-world demo.

## Timing Constraints

- **Bit period:** TH + TL = 1.25 µs ± 600 ns (at 800 Kbps data rate)
- **T0H:** 0.4 µs ± 150 ns; **T0L:** 0.85 µs ± 150 ns
- **T1H:** 0.8 µs ± 150 ns; **T1L:** 0.45 µs ± 150 ns
- **Reset pulse (RES):** ≥ 50 µs low; the transport appends 16 zero-bytes (≈53 µs) automatically
- **Transmission time for n pixels:** n × 24 × 1.25 µs + 50 µs reset ≈ (30n + 50) µs
  - 30 pixels ≈ 950 µs; 300 pixels ≈ 9.05 ms; 1024 pixels ≈ 30.8 ms (≈32 fps)
- **No per-pixel read-back:** the chip is write-only; the driver must maintain its own buffer

## Implementation Notes

- **GRB order:** the datasheet specifies G7…G0, R7…R0, B7…B0 (Green first). The driver must reorder before handing bytes to the transport; the transport itself is order-agnostic.
- **Transport write is blocking and timing-critical:** do not interleave other I/O during a `transport.write()` call. The SPI bit-encoding approach (used by all platforms) is interrupt-safe on most hosts, but bare-metal environments may need interrupts disabled.
- **Buffer ownership:** the chip driver owns the internal GRB buffer. The transport receives an encoded copy; the driver's stored buffer stays in RGB (or raw user values) to allow brightness scaling without precision loss.
- **Brightness scaling at show() time:** storing raw user values and scaling just before transmission avoids accumulated rounding errors if brightness changes frequently.
- **HSV conversion:** for embedded targets without `math` (MicroPython), implement a fixed-point or lookup-table HSV → RGB. For host targets (Linux, Node.js), use the standard library.
- **Pixel count is fixed at construction:** the transport buffer size is `n × 3` bytes. No dynamic resize.
- **`fill()` in Minimal auto-shows:** Minimal calls `transport.write()` inside `fill()`. Full separates buffer mutation from transmission — `set_pixel()` / `set_pixels()` do not call the transport; `show()` must be called explicitly.
- **`fill()` inherited in Full:** Full inherits Minimal's `fill()` as a convenience (fill all + immediate show). This is intentional — it is the fast path for all-same-color updates.
- **Cascade note:** the driver controls all `n` pixels in one strip. Multiple strips on separate SPI buses require separate driver instances.
