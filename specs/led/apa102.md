# Chip Spec: APA102

**Manufacturer:** Shenzhen LED Color Opto Electronic  
**Datasheet:** `datasheets/led/apa102.pdf`  
**Category:** led  
**Transports:** SPI

## Overview

The APA102 is an addressable RGB LED with an integrated constant-current driver IC in a 5050 SMD package. It uses a two-wire synchronous SPI protocol (clock + data) rather than the single-wire NZR protocol of the NeoPixel family. The synchronous clock eliminates all timing constraints, making the APA102 compatible with any SPI bus speed and safe to drive from systems where GPIO timing is non-deterministic (Linux userspace, RTOS with preemption). Each pixel stores a 5-bit global hardware brightness (0–31, scales drive current) plus three 8-bit PWM channels (Blue, Green, Red), giving an effective 13-bit color depth per channel. Pixels cascade via SDI/CKI → SDO/CKO; the host needs only two GPIO or SPI-MOSI/SCK lines.

## Transport Configuration

### SPI

- **Mode:** CPOL=0, CPHA=0 (Mode 0) — data sampled on rising clock edge
- **Bit order:** MSB first
- **Max clock:** up to ~20 MHz in practice; datasheet example uses 512 kHz
- **CS:** not required by the protocol (start frame provides synchronization); if a hardware CS line is available, assert it low for the entire frame (start + pixels + end)
- **Frame structure:** start frame (4 zero-bytes) + n×4 pixel bytes + end frame (≥⌈n/16⌉ bytes of 0xFF, minimum 4 bytes)
- **Max pixels at 30 fps (512 kHz):** 319 — higher SPI clocks increase this proportionally

## Pixel Data Format

No register map. The entire interface is a streaming SPI transaction.

### Frame structure

```
[Start: 0x00 × 4] [LED1: 4 bytes] [LED2: 4 bytes] … [LEDn: 4 bytes] [End: 0xFF × max(4, ⌈n/16⌉)]
```

### LED frame (32 bits, MSB first)

| Bits 31–29 | Bits 28–24             | Bits 23–16    | Bits 15–8     | Bits 7–0     |
|------------|------------------------|---------------|---------------|--------------|
| `111`      | Global brightness (0–31) | Blue (B7–B0) | Green (G7–G0) | Red (R7–R0) |

- First byte: `0xE0 | brightness` — the three high bits are always 1; the lower 5 bits are the hardware brightness
- Wire channel order: BGR (Blue before Green before Red)

### Global brightness (hardware)

| 5-bit value | Drive current |
|-------------|---------------|
| `00000` | 0/31 (off) |
| `00001` | 1/31 |
| `11111` | 31/31 (max, ~24.5 mA) |

### End frame size

⌈n/2⌉ extra clock pulses are needed to propagate data through the full cascade. With 0xFF end bytes (8 pulses each): `max(4, (n + 15) // 16)` bytes of 0xFF.

## Initialization Sequence

1. Construct the SPI transport (Mode 0, MSB first).
2. Construct `APA102Minimal(transport, n)` where `n` is the number of pixels.
3. The constructor allocates an internal buffer of `n × 4` bytes initialized to `[0xE0|31, 0, 0, 0]` per pixel (hardware brightness=31, all LEDs off).
4. No further chip initialization is required.

## Implementation Stages

### Minimal

Goal: light up an entire strip with one color in three lines — construct, fill, done.

| Operation | Parameters | Returns | Notes |
|-----------|------------|---------|-------|
| `init` | `transport`, `n: int` | — | Allocates `n × 4` buffer; per-pixel hardware brightness defaults to 31 |
| `fill` | `r: int`, `g: int`, `b: int` | — | Fills every pixel with (r, g, b) at hardware brightness 31; clamps to [0, 255]; sends immediately |
| `off` | — | — | Equivalent to `fill(0, 0, 0)`; turns the entire strip off |

**Sensible defaults:** Buffer initialized to hardware brightness 31, all channels 0. `fill()` always calls `transport.write()` — there is no separate show step.

### Full

Goal: expose per-pixel addressing, per-pixel hardware brightness, explicit frame control, global software brightness scaling, and convenience helpers. Extends Minimal.

| Operation | Parameters | Returns | Notes |
|-----------|------------|---------|-------|
| *(inherits Minimal)* | | | |
| `set_pixel` | `index: int`, `r: int`, `g: int`, `b: int`, `pixel_brightness: int` (default 31) | — | Write one pixel into the buffer (0-indexed); clamps RGB to [0, 255], brightness to [0, 31]; does **not** send |
| `set_pixels` | `colors: list[tuple]` | — | Each element is `(r, g, b)` or `(r, g, b, pixel_brightness)`; missing brightness defaults to 31; writes into buffer starting at pixel 0; does not send |
| `show` | — | — | Send the full frame (start + buffer with software brightness applied + end) |
| `brightness` | `value: int` (0–255, get/set) | `int` | Global software brightness scalar; default 255; applied to RGB channels at `show()` time: `sent = stored × brightness // 255`; does **not** affect the per-pixel hardware brightness field |
| `rotate` | `steps: int` (default 1) | — | Shift the pixel buffer left by `steps` positions (whole pixels); wraps around; does not send |
| `fill_hsv` | `h: float` (0.0–1.0), `s: float` (0.0–1.0), `v: float` (0.0–1.0) | — | Convert HSV to RGB, fill every pixel at hardware brightness 31, then send immediately |

**Additional configuration options:**
- Per-pixel hardware brightness (0–31) via `set_pixel()` and `set_pixels()`
- Global software brightness scaling (0–255) applied non-destructively at `show()` time
- Explicit frame control: build frames with `set_pixel()` / `set_pixels()`, push with `show()`
- Pixel rotation for chase/scroll effects
- HSV fill for hue-sweep animations

## Data Conversion

```
# User (r, g, b) → wire BGR order
frame_byte_0 = 0xE0 | (pixel_brightness & 0x1F)  # 0b111BBBBB
frame_byte_1 = blue
frame_byte_2 = green
frame_byte_3 = red

# Global software brightness scaling (applied at show() time in Full)
sent_r = stored_r * brightness // 255
sent_g = stored_g * brightness // 255
sent_b = stored_b * brightness // 255

# Full SPI transaction
start_frame = bytes(4)                              # 0x00 × 4
end_frame   = bytes([0xFF] * max(4, (n + 15) // 16))
spi_data    = start_frame + pixel_buffer + end_frame

# HSV → RGB (standard algorithm, all inputs 0.0–1.0, outputs 0–255 int)
# Use platform math library or implement inline for embedded targets.
```

## Examples

### Demo

Demonstrate the APA102's 13-bit effective color depth: run a smooth rainbow sweep for 10 seconds using full hardware brightness (31) and 8-bit RGB values, then repeat the same rainbow at hardware brightness 1 (1/31 current) for 10 seconds to show the difference between software dimming and hardware current control. Print the current mode, hardware brightness, and frame rate once per second. Use 30 pixels, update at approximately 60 fps (16 ms sleep) to show that the APA102's synchronous SPI allows high refresh rates without timing constraints. This exercises per-pixel hardware brightness, global software brightness, the `fill_hsv` method, and explicit `set_pixel` / `show` frame control.

## Timing Constraints

- **None for bit timing:** the synchronous SPI clock eliminates all NZR-style timing requirements.
- **SPI clock:** any clock up to ~20 MHz works in practice; the datasheet example uses 512 kHz.
- **Frame rate:** at SPI clock `F` Hz: max fps = `F / (8 × (4 + 4n + max(4, ⌈n/16⌉)))` where `n` is the pixel count.
  - 30 pixels at 1 MHz: 1,000,000 / (8 × (4 + 120 + 4)) = ~966 fps (far above display needs)
  - 300 pixels at 1 MHz: 1,000,000 / (8 × (4 + 1200 + 19)) = ~101 fps
- **Start frame requirement:** must be exactly 4 zero-bytes; shorter start frames may not reset all ICs in the chain.
- **End frame requirement:** must supply at least ⌈n/2⌉ additional clock pulses; `max(4, (n+15)//16)` bytes of 0xFF satisfies this for any n.

## Implementation Notes

- **BGR wire order:** the datasheet specifies Blue before Green before Red in the LED frame. The driver accepts RGB from the user and reorders internally.
- **LED frame preamble:** first byte of each LED frame must have the three MSBs set to 1: `0xE0 | brightness`. Writing `0xFF` (brightness=31) is valid and simplifies code when per-pixel brightness is not needed.
- **Buffer ownership:** the driver owns an internal buffer of `n × 4` bytes in the same wire layout (already ordered as brightness/B/G/R). Software brightness scaling is applied on-the-fly during `show()`, not stored.
- **Minimal `fill()` auto-shows:** Minimal calls `transport.write()` inside `fill()`. Full separates buffer mutation from transmission.
- **`fill()` inherited in Full:** Full inherits Minimal's `fill()` as a convenience (fill all at brightness=31 + immediate show). This is intentional.
- **CS handling:** the APA102 does not have a hardware CS input — SDI/CKI are continuous. If the SPI peripheral asserts CS, it should span the entire frame (start + pixels + end). Never break the frame across multiple SPI transactions.
- **Software vs hardware brightness:** the global `brightness` property (0–255) scales RGB values in software at `show()` time; it does **not** change the 5-bit hardware brightness field. For visual smoothness at low intensities, use hardware brightness to reduce current (avoids PWM dithering artifacts at 8-bit level) and software brightness for fine control.
- **Pixel count is fixed at construction:** no dynamic resize.
- **rotate() stride:** rotation operates on whole pixels (4-byte units).
- **Cascade:** the driver controls all `n` pixels in one strip. Multiple strips on separate SPI buses require separate driver instances.

## Sigrok Decoder

Two-layer stack: `logic → spi → apa102`.

The `spi` transport decoder (SPI Mode 0, MSB first) captures the raw byte stream and produces `OUTPUT_PYTHON` byte packets for the chip decoder to consume.

The `apa102` chip decoder (id `apa102`, input `['spi']`) consumes bytes and identifies:
- **Start frame:** 4 zero-bytes → annotation "Start"
- **LED frame:** groups 4 bytes; validates that bits 31–29 are `111`; annotates pixel index, hardware brightness (0–31), and R/G/B values (reordered from BGR wire order), plus a `#RRGGBB` hex colour string
- **End frame:** trailing 0xFF bytes → annotation "End (N bytes)"

Emits a warning if an LED frame arrives with incorrect preamble bits. Emits a warning if the end frame is shorter than `(n+15)//16` bytes for the observed pixel count.
