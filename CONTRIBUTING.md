# Contributing

This project is AI-implemented: chip drivers and infrastructure are generated by Claude Code and OpenCode. Human contributors are welcome to add chips, fix bugs, or improve documentation using the same workflow.

## Quick orientation

| File | Purpose |
|------|---------|
| `CLAUDE.md` | Repo layout, category list, Minimal/Full pattern, AI workflow |
| `AGENTS.md` | Code conventions and patterns (written for AI, useful for humans too) |
| `TESTING.md` | How to run hardware tests on all platforms |
| `specs/<category>/<chip>.md` | Single source of truth for each chip driver |
| `specs/_template_chip.md` | Template for new chip specs |

## Adding a chip

### 1. Get the datasheet

Download the datasheet PDF and commit it:

```
datasheets/<category>/<chip>.pdf
```

Categories are listed in `CLAUDE.md`. Use the closest fit; `other` is a valid fallback.

### 2. Write the spec

Copy `specs/_template_chip.md` to `specs/<category>/<chip>.md` and fill it in completely:

- Register map with bit fields
- Initialization sequence and timing
- Data conversion formulas
- Minimal and Full API tables (with all defaults for Minimal explicit)
- Demo scenario description

The spec is the reference documentation. Code is derived from it.

### 3. Implement

Create a feature branch (`feature/<chip>`) and implement against the spec:

| Language | Paths |
|----------|-------|
| Python driver | `python/periph/chips/<category>/<chip>.py` |
| Python examples | `python/examples/<category>/<chip>/minimal.py`, `complete.py`, `demo.py` |
| C++ driver | `cpp/src/chips/<category>/<Chip>.h`, `<Chip>.cpp` |
| C++ examples | `cpp/examples/<Chip>_Minimal/`, `<Chip>_Complete/`, `<Chip>_Demo/` (dir name = .ino name) |
| Node.js driver | `nodejs/packages/periph/src/chips/<category>/<chip>.js` |
| Node.js examples | `nodejs/packages/periph/examples/<category>/<chip>/minimal.js`, `complete.js`, `demo.js` |
| Node-RED node | `nodejs/packages/node-red-contrib-periph-<category>/nodes/<chip>/<chip>.js`, `<chip>.html` |
| Node-RED flow | `nodejs/packages/node-red-contrib-periph-<category>/examples/<chip>/demo.json` |

When adding the first file to a directory that contains only `.gitkeep`, remove the `.gitkeep`.

See `AGENTS.md` for detailed code conventions — class structure, transport interface, language-specific rules.

### 4. Write tests

Add one test file per platform following the naming convention in `TESTING.md`. Use `INA226` as the reference.

### 5. Open a PR

- One commit per implementation stage (Minimal, then Full)
- Link the datasheet in the PR description
- CI must pass (compile checks on Arduino, Linux GCC, Python syntax, Node.js syntax)

## Code conventions (summary)

Full detail is in `AGENTS.md`. The key rules:

**All platforms**
- Drivers depend only on the transport interface — never on a concrete transport class
- Two classes per chip: `<Chip>Minimal` (base) and `<Chip>Full` (extends Minimal, never duplicates)

**Python**
- No f-strings, walrus operator, or `match` — MicroPython lags CPython
- No type annotations
- Constants as `_REG_NAME = 0x00` class-level variables

**C++ (Arduino)**
- No STL, no exceptions, no heap allocation in drivers
- Register constants as `static constexpr uint8_t`
- CommonJS only (`require` / `module.exports`) — required for Node-RED

**Node.js**
- CommonJS only (`require` / `module.exports`)
- camelCase methods, UPPER_SNAKE constants

## Running tests

See [TESTING.md](TESTING.md).

Compile-only checks (no hardware needed):

```
cpp/test_arduino.sh  --compile-only power/ina226
cpp/test_linux.sh    --compile-only power/ina226
```

## Reporting issues

Open a GitHub issue. Include the chip name, platform, and a minimal reproduction if it's a driver bug.
